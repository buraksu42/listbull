## Domain instance

> Pattern type from Stack Defaults: "Main flagship + experiments subdomain".
> Mode here: **flagship under umbrella** — `listbull.org` is the umbrella; the
> listgram app deploys to subdomains.

### 3-tier domain structure

`listbull.org` is the umbrella domain. Three categories of host:

1. **Apex `listbull.org`** — open-source project info / install docs site. Static, decoupled from this codebase. Hosted as a separate deliverable (Cloudflare Pages / GitHub Pages / dedicated container). Contains the pitch, GitHub link, install guide, contributing guide, changelog. **NOT the listgram app.** See `docs/launch-checklist-phase-5.md` § Step 11 for hosting options.
2. **Canonical app deployment `prod.listbull.org`** — production listgram instance (Mini App + bot + DB). This codebase, prod Hetzner server.
3. **Test/staging `test.listbull.org`** — same code as prod, lower stakes; deploys on every `dev` push.
4. **Tenant pattern `<tenant>.listbull.org`** — additional self-host instances on the same infra (e.g. `loyetta.listbull.org`). Same code, separate `NEXT_PUBLIC_APP_URL`, `DATABASE_URL`, `TELEGRAM_BOT_TOKEN`. Sharing or dedicated server depending on scale.

### URLs / endpoints for the listgram app

- **Prod URL**: `https://prod.listbull.org`
- **Test/staging URL**: `https://test.listbull.org`
- **Bot username**: `@listgram_bot` for prod (must be reserved via BotFather pre-deploy; fall back
  to `@listgram_app_bot` if taken). Test deployment uses a separate bot (e.g. `@listgram_test_bot`). Tenants pick their own bot per deployment.
- **Mini App URL**: `https://prod.listbull.org/app` (linked from `BotFather` Mini App config). Tenant deployments use `https://<tenant>.listbull.org/app`.
- **DNS**: Cloudflare —
  - `prod.listbull.org` → `46.224.144.255` (Hetzner prod), proxy OFF (Let's Encrypt HTTP-01 needs direct origin)
  - `test.listbull.org` → `62.238.8.55` (Hetzner test), proxy OFF
  - `listbull.org` (apex) → static project info site host (per launch checklist Step 11), proxy OK
  - `<tenant>.listbull.org` → per-tenant server, proxy OFF (same webhook constraint)
- **Apex behavior**: `listbull.org` is a **distinct site** (project info + install docs), NOT a redirect to the app. Visitors arriving at `listbull.org` are looking for "what is this and how do I install it"; visitors arriving at `prod.listbull.org` are looking for "open in Telegram". The two surfaces have different audiences.
- **Cert mechanism**: hosting-managed (Dokploy / Traefik handles Let's Encrypt for app subdomains; apex site host handles its own).
- **Bot webhook URL**: `https://prod.listbull.org/api/telegram/webhook` (prod),
  `https://test.listbull.org/api/telegram/webhook` (test); per-tenant bots use
  `https://<tenant>.listbull.org/api/telegram/webhook`.
- **Status**: DNS pending (post-deploy step)

## Stack overrides

None. Standard Stack Defaults apply: Next.js + Tailwind + shadcn/ui + Postgres + Drizzle
+ Better Auth + Resend + OpenRouter + Sentry on Hetzner+Dokploy.

**Project-specific additions** (these ARE inside Stack Defaults' "Extras" allowance, not
overrides):

- `grammY` — Telegram bot framework (Node.js, modern, full TypeScript types). Chosen
  over Telegraf because grammY's middleware model maps cleanly to Next.js route handler
  pattern; `@grammyjs/types` gives typed `Update` payload.
- `@telegram-apps/sdk-react` — official Telegram Mini App SDK (theme params, initData,
  haptics, MainButton, BackButton, full-screen mode).
- `next-intl` — i18n (TR + EN at launch; framework-native for App Router).
- `@anthropic-ai/sdk` (used through OpenRouter base URL) — LLM client. Anthropic SDK
  preferred over OpenAI SDK because Anthropic's tool-calling format is what we'll use
  in production; with `baseURL` swap, OpenRouter routes to either Anthropic or other
  models seamlessly. Default model: `anthropic/claude-sonnet-4`.

## Database schema

PostgreSQL 16 (per Stack Defaults); migrations via Drizzle. All tables include
`created_at` and `updated_at` (timestamptz, default `now()`) unless noted. UUIDs are
`uuid` type, default `gen_random_uuid()`.

### `users`

- `id` (uuid pk)
- `telegram_id` (bigint unique not null) — Telegram numeric user ID
- `telegram_username` (text nullable) — user's `@username`; can change over time
- `telegram_first_name` (text not null)
- `telegram_last_name` (text nullable)
- `telegram_photo_url` (text nullable)
- `locale` (text not null default `'en'`) — `'tr'` or `'en'`
- `timezone` (text not null default `'UTC'`) — IANA name; user-editable in Mini App
- `openrouter_api_key_encrypted` (text nullable) — BYOK; AES-256-GCM with `ENV_KEY`
- `llm_model` (text not null default `'anthropic/claude-sonnet-4'`)
- `notifications_enabled` (boolean not null default `true`)
- `created_at`, `updated_at`

**Indexes**: `users_telegram_id_idx` (unique on `telegram_id`),
`users_telegram_username_idx` (lower-cased on `telegram_username`).

**Notes**: `telegram_username` deliberately not unique — multiple historical users may
hold the same handle over time. Resolve invitations against current value at invite-create
time, store immutable user-id reference once accepted.

### `lists`

- `id` (uuid pk)
- `name` (text not null)
- `emoji` (text nullable) — single emoji shown next to list name
- `owner_id` (uuid not null, fk `users.id` on delete cascade)
- `is_inbox` (boolean not null default `false`) — exactly one inbox list per user, auto-created on `/start`
- `archived_at` (timestamptz nullable) — soft delete; Phase-2 RAG hook
- `created_at`, `updated_at`

**Indexes**: `lists_owner_id_idx`, partial unique
`lists_owner_inbox_unique` on `(owner_id) where is_inbox = true`.

### `list_members`

- `id` (uuid pk)
- `list_id` (uuid not null, fk `lists.id` on delete cascade)
- `user_id` (uuid not null, fk `users.id` on delete cascade)
- `role` (text not null) — enum-via-check: `'owner' | 'editor' | 'viewer'`
- `invited_by` (uuid nullable, fk `users.id`)
- `accepted_at` (timestamptz not null default `now()`) — owner row gets list creation time
- `created_at`, `updated_at`

**Indexes**: unique `list_members_list_user_unique` on `(list_id, user_id)`,
`list_members_user_id_idx` on `user_id` (lookup "all my lists").

**Notes**: owner role is auto-inserted by `lists` create transaction; never modified
directly. Removing the owner deletes the list (cascade). Editors can mutate items;
viewers read-only (Phase 1 implements editor + owner only; viewer reserved for Phase 2).

### `items`

- `id` (uuid pk)
- `list_id` (uuid not null, fk `lists.id` on delete cascade)
- `text` (text not null) — the item content
- `is_checkable` (boolean not null default `true`) — todo if true, note if false
- `is_done` (boolean not null default `false`)
- `assignee_id` (uuid nullable, fk `users.id` on delete set null)
- `due_at` (timestamptz nullable) — scheduled reminder time
- `reminder_sent` (boolean not null default `false`)
- `position` (integer not null default `0`) — for drag-reorder; sparse-numbered
- `created_by` (uuid not null, fk `users.id`)
- `completed_at` (timestamptz nullable)
- `archived_at` (timestamptz nullable) — Phase-2 RAG hook
- `created_at`, `updated_at`

**Indexes**:
- `items_list_id_idx` on `(list_id, archived_at, is_done, position)` — main list view
- `items_due_at_idx` on `due_at` partial `where due_at is not null and reminder_sent = false`
  — cron job pickup
- `items_assignee_idx` on `(assignee_id, is_done)` — "items assigned to me"

### `messages`

- `id` (uuid pk)
- `user_id` (uuid not null, fk `users.id` on delete cascade)
- `chat_id` (bigint not null) — Telegram chat_id
- `role` (text not null) — `'user' | 'assistant' | 'tool'`
- `content` (text not null) — message text or tool result JSON
- `tool_calls` (jsonb nullable) — array of `{ id, name, input }` when role='assistant'
- `tool_call_id` (text nullable) — when role='tool', references the assistant message's tool call
- `created_at` (no `updated_at`; messages are append-only)

**Indexes**: `messages_chat_recent_idx` on `(user_id, chat_id, created_at desc)` —
hot path for context window assembly.

### `list_invites`

- `id` (uuid pk)
- `list_id` (uuid not null, fk `lists.id` on delete cascade)
- `invited_username` (text not null) — Telegram `@username` lower-cased
- `invited_by` (uuid not null, fk `users.id`)
- `token` (text unique not null) — random 32-byte URL-safe; deeplink param
- `role` (text not null default `'editor'`) — what role they'll get on accept
- `expires_at` (timestamptz not null) — default 7 days from creation
- `accepted_at` (timestamptz nullable)
- `accepted_by_user_id` (uuid nullable, fk `users.id`)
- `created_at`

**Indexes**: `list_invites_token_idx` (unique), `list_invites_list_idx`.

### `activity_log`

Single dual-purpose table: user-facing feed (B1) AND audit/restore log (F2).

- `id` (uuid pk)
- `list_id` (uuid nullable, fk `lists.id` on delete cascade)
- `entity_type` (text not null) — `'item' | 'list' | 'member'`
- `entity_id` (uuid not null) — id of affected entity (may be soft-deleted)
- `action` (text not null) — see action enumeration below
- `actor_id` (uuid not null, fk `users.id`)
- `payload_before` (jsonb nullable) — full state before change (for restore)
- `payload_after` (jsonb nullable) — full state after change (for diff display)
- `created_at`

**Indexes**:
- `activity_list_recent_idx` on `(list_id, created_at desc)` — feed query
- `activity_entity_idx` on `(entity_type, entity_id, created_at desc)` — restore lookup

**Action enumeration** (text values, no enum type for migration ease):
`item_created`, `item_completed`, `item_uncompleted`, `item_edited`, `item_deleted`,
`item_assigned`, `item_unassigned`, `item_due_set`, `item_due_cleared`,
`list_created`, `list_renamed`, `list_archived`, `list_restored`,
`member_added`, `member_removed`, `member_role_changed`.

**Restore window**: items deleted ≤30 days ago can be restored by reading
`payload_before` from the corresponding `item_deleted` row. Older entries are kept (for
audit) but UI does not surface them.

## Auth

- **Library:** Better Auth (Stack Defaults' choice) for Mini App session management.
- **Telegram-specific layer:** custom HMAC-SHA256 verification of `initData` per the
  official Telegram WebApp spec, mounted as a Better Auth plugin in `src/lib/auth/telegram-plugin.ts`.
- **Bot side:** no Better Auth — bot trusts Telegram's signed `Update` payload (we verify
  webhook secret token in header). User lookup by `from.id` (`telegram_id`).
- **Providers:** Telegram only (no email/password, no magic link, no OAuth). Self-hosted
  forks may add other providers; default install is Telegram-only.
- **Session:** cookie-based (Better Auth default) for Mini App; bot is stateless, derives
  user from each update.
- **Features:** none of email-verify / 2FA / password-reset apply (no password). Account
  creation is auto-on-/start.

## Email

- **Provider:** Resend (Stack Defaults).
- **Sending domain:** `noreply@listbull.org` (DKIM/SPF on Cloudflare DNS).
- **Email types:** none in Phase 1. **Drop section in CLAUDE.md.**

**Rationale:** all notifications go via Telegram DM (the bot already has the chat).

## AI / LLM

- **Provider:** OpenRouter (Stack Defaults).
- **Default model:** `anthropic/claude-sonnet-4`. Per-user override available in Mini App
  settings (preset list: `anthropic/claude-haiku-4.5`, `openai/gpt-4o-mini`,
  `anthropic/claude-sonnet-4`, `anthropic/claude-opus-4.7`). User's choice persists in
  `users.llm_model`.
- **Auth model:** **BYOK**. User pastes their OpenRouter API key into Mini App settings
  (encrypted at rest in `users.openrouter_api_key_encrypted`). All LLM calls use the
  user's key, not the host operator's. If no key is set, bot replies: "Add your
  OpenRouter API key in Mini App settings to use AI features."
- **UI mode:** non-streaming for bot replies (Telegram doesn't support streaming text
  edits at sub-second cadence; we send full reply once tool round-trips complete).
  Streaming reserved for Mini App "ask listgram" experimental Phase-2 feature.
- **Token budget per user:** default no app-side cap (BYOK = user pays). Optional
  env-configurable `LISTGRAM_PER_USER_HOURLY_MSG_LIMIT` for self-host operators worried
  about runaway costs (default `0` = disabled).
- **Tool calls:** 9 tools defined in `src/lib/ai/tools.ts`:
  `create_item`, `search_items`, `update_item`, `delete_item`, `complete_item`,
  `list_lists`, `share_list`, `schedule_reminder`, `assign_item`. Tool definitions
  include zod schema for validation; tool execution wraps in Drizzle transaction +
  activity_log write.
- **Embeddings:** none in Phase 1. Phase-2 RAG over archived items will use pgvector.

## i18n

- **Locales:** `tr`, `en`. User's locale is initialized from Telegram `language_code` on
  `/start`; user can override in Mini App settings.
- **Library:** `next-intl` (Next.js App Router native).
- **Routing:** **no path prefix.** Mini App reads locale from `users.locale` (server-side
  per-request); routes stay clean (`/app/lists`, not `/tr/app/lists`). Bot replies use
  `users.locale` directly. URL prefixes are useful for SEO; Mini App is auth-gated and
  bot replies are private DMs — neither needs SEO routing.
- **Locale-aware formatting:** yes — date/time via `Intl.DateTimeFormat` honoring
  `users.timezone` and `users.locale`. Specifically: due-date display, activity log
  timestamps, scheduled reminder copy.
- **LLM language:** system prompt instructs the model to respond in user's locale; for
  mixed-language input, model auto-detects dominant language. (E3 enhancement.)

## Performance considerations

- **Bot webhook latency target:** ≤2s for non-LLM responses, ≤8s when LLM tool round-trip
  involved. Telegram allows 60s but UX feels sluggish past 5s. If LLM exceeds 8s, send
  intermediate `chatAction: 'typing'` status; reply when complete.
- **Mini App initial load:** target First Contentful Paint <1.5s on 4G. Mini App lives
  under `(app)` route group with minimal layout — no marketing chrome bundled.
  Telegram's WebView caches aggressively.
- **List view rendering:** virtualize lists >100 items (TanStack Virtual). Phase 1 most
  users will not hit this; the cap is wired for Phase 2 RAG'd "all completed items"
  views.
- **Conversation history context window:** cap at last 30 messages or ~6k tokens
  (whichever first). Older messages not loaded into LLM context but remain in DB.
- **Activity log growth:** active list with two collaborators may generate ~50 rows/week.
  Year-1 ceiling per list ~3k rows; partition not needed at Phase-1 scale. Revisit if
  any one list crosses 50k rows.
- **Webhook throughput:** Telegram retries on 5xx with exponential backoff; if bot
  process restarts, queue can build up. Webhook handler must respond 200 within 60s
  even if LLM is slow — pattern: ack 200 immediately, do LLM work in background,
  then `sendMessage`. (Webhook handler returns 200 + Promise.resolve; deferred work
  uses `waitUntil` if available, else `setImmediate`.)

## Monitoring (project-specific add)

- **Health endpoint:** `GET /api/health` → `{"status":"ok","db":"ok","ts":<ms>}` (validate
  Postgres connectivity with `SELECT 1`; uptime monitor matches `"status":"ok"`).
- **Cron heartbeat:** scheduler logs to a `heartbeats` row each minute (or pings
  Better Stack heartbeat URL); if missed >5 min, alert. (Optional, env-configurable
  `LISTGRAM_HEARTBEAT_URL`.)
- **Release tagging in Sentry:** Dokploy post-deploy hook calls Sentry's release CLI
  with the merged commit SHA — same pattern as SocialCal.
- **Bot-specific monitor:** Telegram webhook dashboard (BotFather `/setwebhook` info)
  shows pending update count + last error. Low-tech but free.

## Project-specific gotchas

- **Bot username squatting.** `@listgram_bot` may be taken; check via
  `https://t.me/listgram_bot` and `BotFather → /mybots` before deploy. If taken, fall
  back to `@listgram_app_bot` and update README + Mini App config.
- **Webhook secret token rotation.** Telegram allows a `secret_token` header on webhook
  requests (set via `setWebhook`). Verify on every request; rotate on prod via env
  redeploy. Failing to verify = open webhook = anyone can spoof updates.
- **initData verification edge cases.** Mini App `initData` query string is signed; expires
  in 24h per Telegram convention. Re-issue session cookie on every Mini App load when
  initData is fresh, fall back to existing session if expired (Better Auth handles).
- **Telegram message length cap.** Outgoing messages capped at 4096 chars. LLM replies
  occasionally exceed; chunk on word boundaries before sending.
- **Markdown escaping.** Telegram MarkdownV2 requires escaping `_*[]()~\`>#+-=|{}.!`.
  Use grammY's `formatter` helper, never raw concat.
- **Cron timezone.** Dokploy cron container runs UTC. Reminder due-time comparisons must
  use UTC consistently; user's local time is presentation-only.
- **Self-host operators are data controllers.** README must include "data flows" section
  + GDPR-relevant notes (where Postgres lives, retention policy = forever unless user
  exports + deletes).

## Work plan & agent architecture

### Execution mode

**Multi-agent (hybrid)** — single session for Phase 1 (Foundation), multi-agent from
Phase 2 onward. Stack Defaults Working pattern: "Hybrid (orchestrator + sub-agents
for Full mode projects)" matches.

### Phases

5 phases. Each scoped to one verification gate.

- **Phase 1 — Foundation**
  - Scope: scaffold Next.js + Drizzle migrations + `users` + `lists` + `list_members` +
    `items` + `messages` + `activity_log` + `list_invites` schema; webhook endpoint
    skeleton; Mini App layout shell; initData verify; Telegram-native theme adapter;
    /start handler creates user + Inbox; `/help`, `/lists`, `/start` slash commands;
    Mini App lists overview + item list view (read-only).
  - Verification: pragmatic gate — `npm run lint`, `tsc --noEmit`, smoke: send `/start`
    to test bot, see Inbox auto-created in DB, open Mini App, see Inbox list rendered.

- **Phase 2 — Core conversational + manage**
  - Scope: LLM tool layer (`create_item`, `search_items`, `update_item`,
    `complete_item`, `delete_item`, `list_lists`); Mini App item check/uncheck/edit/
    delete + drag reorder; conversation history table writes; bot conversational
    capture + recall + general Q&A; multi-turn; BYOK key entry in Mini App settings.
  - Verification: pragmatic gate + smoke: send "süt al" to bot, item appears in
    Inbox; ask "okuma listemde ne var", LLM calls search_items, returns natural reply;
    toggle item in Mini App, change reflects in DB; settings save BYOK key.

- **Phase 3 — Sharing + reminders + assignments**
  - Scope: `share_list` tool + `/share` slash command; `list_invites` flow with
    deeplinks; Mini App share UI + member management; `assign_item` tool + @mention
    parsing; `schedule_reminder` tool; cron job runner; activity_log writes from all
    mutations; B1 activity feed view.
  - Verification: pragmatic + smoke: invite second test user via deeplink, accept,
    cross-account item creation works; @mention assignment writes assignee_id;
    scheduled item triggers DM at due_at within 1min.

- **Phase 4 — Polish + open-source quality**
  - Scope: D1 inline mode; D2 shareable list snapshot; D3 schedule-a-message pickup;
    A3 forwarded message handling; F1 data export; F2 audit log restore; E1+E2+E3
    i18n + Mini App accessibility + bot multilingual; Docker Compose deploy file;
    `.env.example` + first-run setup wizard; README; LICENSE (MIT); CONTRIBUTING.
  - Verification: strict gate (final phase) — full Playwright E2E suite passes,
    Lighthouse a11y ≥95 on Mini App, Docker Compose `up` works on clean machine.

- **Phase 5 — Launch prep**
  - Scope: production DNS cutover, BotFather Mini App config, robots.txt for prod
    (`Allow: /` for marketing landing, `Disallow: /app/` for Mini App), test
    `noindex` confirmation, GitHub repo public + topics, demo video.
  - Verification: smoke from a fresh Telegram account on prod bot; basic flows work.

### Sub-agents

Multi-agent from Phase 2 onward. Roster trimmed to listgram's actual surface (full
spec in `agents.md`):

- **Architect-agent** — folder structure validation, schema review, type freezing in
  `src/lib/types/`, design.md token mapping when phases ship UI bundles.
- **Backend-agent** — bot webhook handler, slash commands, LLM tool implementations,
  Drizzle queries, migrations, cron job, activity_log writes.
- **Frontend-agent** — Mini App routes, components, forms, optimistic UI, theme
  adapter, accessibility.
- **AI-agent** — system prompt, tool schema definitions, multi-turn flow, conversation
  history slicing, streaming UI for Phase-2 experiment, prompt versioning.
- **Reviewer-agent** — per-phase lint/tsc/test runs, security pass on initData / webhook
  secret / encryption-at-rest, OSS-quality pass (README, LICENSE, .env.example).

(5 agents total. No DB-agent — schema is small enough that Backend-agent owns
migrations directly. No DevOps-agent — Dokploy is well-trodden in this stack; CI is
standard.)

### Verification gate style

**Pragmatic** (Stack Defaults default). Phase 4 escalates to **strict** because that's
the OSS-readiness phase and quality bar must be high before public release.

- **Pragmatic** (Phases 1, 2, 3, 5): lint + tsc clean, smoke test the new feature, commit.
- **Strict** (Phase 4): lint + tsc + unit tests on new lib code + Playwright E2E on bot
  webhook + Mini App auth + Lighthouse a11y ≥95 + Docker Compose works on clean machine.

### Coordination notes

- **Phase 2 Backend ↔ AI dependency.** Tool implementations (Backend) and tool schemas
  (AI) ship in lockstep; AI-agent freezes schema first via Architect-agent's review,
  then Backend implements. Mismatch = Phase 2 verification failure.
- **Phase 3 Frontend ↔ Backend.** Mini App share UI consumes
  `POST /api/lists/:id/invite` (Backend); Frontend cannot start until Backend's API
  contract is in `src/lib/types/`. Architect-agent gates the contract freeze.
- **Phase 4 OSS quality.** Reviewer-agent owns Docker Compose validation: spin up on
  a clean Hetzner node, no .env, follow README, confirm Mini App + bot work end-to-end.

## Open architecture questions

- **Webhook vs long-polling for self-host.** Phase 1 uses webhook (requires public
  HTTPS). Self-hosters behind NAT may prefer long-polling. README documents webhook
  as primary; long-polling toggle deferred to Phase 6+ if user demand exists.
- **Per-user OpenRouter key validation timing.** Validating the key on save means
  one extra OpenRouter API call per save. Phase 1 implements lazy validation (validate
  on first LLM call, surface error to user). Could move to eager validation in Phase 4
  if UX feedback says "users don't realize the key is wrong until they ask the bot."
- **Mini App full-screen mode.** Telegram supports it (`requestFullscreen()`); not
  enabled in Phase 1 (default-presented mode is more familiar). Revisit per-user
  setting in Phase 2.
