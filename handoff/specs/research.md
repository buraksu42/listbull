# Research Brief — listbull

> Generated by vibe-researcher on 2026-05-01. This file is the foundation document for
> vibe-architect, vibe-engineer, and vibe-designer.

## 1. Idea snapshot

listbull is a Telegram-native AI assistant with persistent shared list memory. Users chat
with the bot in natural language to capture, manage, and share to-dos & notes; a Telegram
Mini App provides visual management. Open source, self-hostable, BYOK (bring-your-own
OpenRouter key).

Primary persona: power Telegram user (Turkish/English, mobile-first) who wants
zero-friction list management and an AI chat companion in the same surface they already
use 50× a day. Secondary: their household members and close collaborators with whom
specific lists are shared.

## 2. Competitive landscape

| Competitor | Strength | Weakness | Pricing | Notes |
|------------|----------|----------|---------|-------|
| **listOK** (`@list_ok_bot`) | Smart-create + history dedupe; multi-language; group support | Inline-keyboard UX only; no Mini App; no AI; no granular sharing | Free | Most active "real" todo bot; built by single dev (ShallowDepth) |
| **ToBeDo** (`@tobedo_bot`) | Group-message-to-checklist auto-conversion is a clever pattern | Group-as-list (no granular sharing); minimal feature set; OSS but unmaintained | Free / OSS | devforth/tobedo on GitHub, dockerized but stale |
| **Taskobot** (`@tasko_bot`) | Inline mode for collaborative task assignment in chats | Single-task assignment, not list management; tiny user base | Free | Niche pattern: 1:1 task confirmation flow |
| **todobbot** (`@todobbot`) | Simple group todo lists | Abandoned (~2018-2022 era); inline keyboard UX | Free | Discoverable but stale |
| **AnyList / OurGroceries** (mobile apps, not Telegram) | Mature shared-list UX (esp. household groceries) | Not Telegram-native; require app install; no AI | Free / $5–10/yr Pro | The "real" competitor for the household sharing use case |

**Default option in space:** listOK — only seriously-maintained Telegram todo bot with
thoughtful UX (smart dedupe, recurring items via syntax). It is what people land on after
trying 2-3 abandoned ones.

**Premium option:** none. There is no "premium" Telegram-native list product. AnyList /
OurGroceries dominate the mobile-app side, but inside Telegram itself the segment is
flat — every option is free, basic, and either abandoned or hobby-tier.

## 3. Market gap

- **Granular per-list sharing.** Existing bots are either single-user or group-as-list.
  No clean Apple-Reminders-style pattern of "share this specific list with this specific
  person, not the whole group." This is the single biggest unmet need across category.
- **Conversational LLM + persistent state.** No Telegram bot today blends free-form
  natural-language interaction with persistent list management. ChatGPT-bot wrappers
  exist but have no list memory; todo bots exist but use rigid commands. Combining the
  two creates a category-of-one inside Telegram.
- **Modern Mini App UI.** Existing players use 2018-era inline keyboards. Telegram Mini
  Apps got dramatic 2024–2026 platform upgrades (full-screen, home-screen shortcuts,
  share flows, secure storage) that nobody in this niche is using.
- **Open-source, self-hostable, modern stack.** Existing OSS Telegram todo bots are
  Python 2 era or unmaintained. There's no reference implementation showing "this is how
  you build a Telegram bot + Mini App in 2026 with TypeScript + Postgres."

**Wedge angle:** First conversational AI list assistant fully native to Telegram —
chat IS the interface, with rich Mini App for management, granular per-list sharing,
and zero install / zero account creation friction.

## 4. Proven technical patterns to replicate

- **Telegram initData HMAC-SHA256 auth** (Telegram official): Mini App receives signed
  user payload from Telegram client; backend verifies signature with bot token. Zero
  password, zero OAuth roundtrip. Reference: core.telegram.org/bots/webapps.
- **Webhook-based bot ingest** (every modern bot): set webhook to HTTPS endpoint, receive
  POST per update, respond 200 fast (<3s). Telegram retries on 5xx. Pattern used by every
  serious Node.js bot.
- **Hybrid bot + Mini App** (DurgerKingBot, BlumApp, Notcoin, official sample): bot
  handles commands and notifications; Mini App handles rich UI. Both share auth via
  initData. This is the proven 2024–2026 architecture for any Telegram product beyond
  trivial.
- **Tool calling for LLM-mediated state ops** (Claude/GPT 2024+): expose `create_item`,
  `search_items`, etc. as tools; let LLM choose when to call. Cleaner than prompt-engineered
  intent classifiers. OpenRouter exposes both Anthropic and OpenAI tool-calling APIs.
- **Soft-delete + activity log dual-purpose table** (every "with restore" SaaS): single
  events table captures both human-readable action (for feed) and full payload (for
  restore). Saves a join + a table.

## 5. Anti-patterns to avoid

- **Group-message-as-list (ToBeDo pattern).** Looks clever in demo, terrible in practice:
  every state change is a noisy chat message; no granular sharing; lists become orphaned
  when group messages scroll away.
- **Hard-coded slash-command UX (~80% of existing bots).** `/add`, `/list`, `/done <n>` —
  feels obsolete next to LLM-mediated UX. Reserve slash commands for deterministic ops
  (`/share`, `/lists`, `/start`, `/help`) only.
- **Redirecting to external website for management** (some legacy bots): breaks the
  "stay in Telegram" promise, kills the user-acquisition wedge. Mini App is the correct
  modern answer.
- **Inline keyboards as primary UX for >10 items.** Telegram caps polls at 10 options;
  inline keyboards become unwieldy at scale. Use Mini App for any list management beyond
  trivial.
- **Hosting bot state in frontend (localStorage / Mini App init data only).**
  Documented failure mode: users switch devices, state vanishes. Backend owns state;
  frontend renders. (Source: nadcab.com gaming Mini App architecture mistake.)

## 6. Top user pain points (with sources)

1. **No granular sharing** — listOK Telegram support thread: users repeatedly ask
   "can I share this list with just my partner, not the whole group?" Answer: no.
   *(shallowdepth.online/projects/listok comments thread)*
2. **Group-bot adds noise to chats** — Reddit r/telegram threads on bot-in-group UX.
   Users report "every action becomes a message; we end up muting the bot." Group-shared
   lists vs DM-shared lists is a UX dichotomy nobody resolves cleanly.
   *(github.com/python-telegram-bot/python-telegram-bot/discussions/2751)*
3. **Old bots rotting** — Hacker News and Reddit comments on Telegram bot recommendations
   typically end with "but I think it's abandoned." Users wary of investing data in
   short-lived bots. Open-source self-hostable matters because of this.
4. **No AI integration in any todo/note bot** — search "telegram bot ai todo"
   returns either ChatGPT wrappers (no list memory) or todo bots (no AI). Users asking
   for both in one place have no answer.
5. **Mobile-first management is awkward** — managing 50+ items across multiple lists via
   inline keyboards is painful on phones. Mini App with proper drag/select/bulk ops is
   the answer; nobody has shipped it well in this niche.

## 7. Pricing & positioning landscape

- **Typical tiers in category (Telegram bots):** virtually all free; sustainability
  models are donations, ads (rare), or abandonment. AnyList-tier mobile apps charge
  $5–10/year for premium (sharing, sync). No Telegram bot has cracked monetization for
  todo/list use case.
- **Free tier wedge:** total free with usage cost externalized to user (BYOK API key)
  is the cleanest play. Users running GPT-class wrappers are already used to bringing
  their own key; this segment is comfortable with pattern.
- **Positioning angles in market:**
  - "Open-source X" — strong fit for OSS-curious dev/power-user audience
  - "Telegram-native X" — strong fit for users who live in Telegram (CIS markets, Türkiye,
    Iran, parts of EU)
  - "AI-native X" — fresh; nobody is doing AI + persistent list memory in Telegram
  - "Privacy-first X" — self-hostable angle resonates with this audience
- **Recommended slot:** "the open-source, AI-native, Telegram-first task & note assistant
  you self-host with your own API key." Combines all four angles above without leaning
  on price.

## 8. Key insights for the specialists

> Direct guidance for vibe-architect / vibe-engineer / vibe-designer.

### For vibe-architect

- **Telegram bot webhook + Mini App = two surfaces, one Postgres.** Single Next.js app
  hosts both: `POST /api/telegram/webhook` for bot updates, `app/(app)/...` routes for
  Mini App. They share Drizzle queries, auth (verified initData), tools.
- **LLM tool calling is the intent layer.** Don't write a custom intent classifier.
  Define ~8 tools (`create_item`, `search_items`, `update_item`, `delete_item`,
  `list_lists`, `share_list`, `schedule_reminder`, `assign_item`); let OpenRouter-routed
  LLM call them. Default model: `anthropic/claude-sonnet-4` (per-user override possible
  via Mini App settings).
- **Conversation history is bounded; archive aggressively.** Keep last ~30 messages per
  chat in LLM context window; older messages live in `messages` table but aren't injected.
  Token costs scale with users; BYOK pushes the cost to user, but UX still benefits from
  keeping context focused.
- **Reminders need cron.** Use Dokploy cron container running once/minute, query
  `items WHERE due_at <= now() AND reminder_sent = false`, send DM, set `reminder_sent`.
  No queue needed at this scale; Postgres is the queue.
- **Soft-delete is the Phase-2 hook.** Schema includes `archived_at`, `completed_at`
  on items; `archived_at` on lists. Phase 1 doesn't query these for RAG; Phase 2 will.
  Cost today: zero.
- **Multi-tenancy is per-user-via-telegram_id, not per-org.** No tenant_id column needed.
  Lists are owned by users; sharing is via `list_members` join table.

### For vibe-engineer

- **Mini App = same Next.js app, different layout.** Mini App routes live in
  `src/app/(app)/...` with a layout that loads `@telegram-apps/sdk-react`. Bot webhook
  is a route handler at `src/app/api/telegram/webhook/route.ts`. Same Drizzle, same
  validators, same lib.
- **Optimistic UI is non-optional in Mini App.** Toggle item, drag-reorder, bulk-select
  are all instant-feel mutations. Use `useOptimistic` (React 19) or TanStack mutate;
  reconcile on server response.
- **Polling 5s for shared list state.** Day-1 sync is polling; revisit websockets in
  Phase 2. TanStack Query with `refetchInterval: 5000` on shared list views; pause
  when tab not active.
- **Tool execution must be transactional.** When LLM calls `create_item`, the handler
  runs INSIDE a single Drizzle transaction: insert item, insert activity_log row,
  return. Half-applied state breaks the audit log.
- **i18n via next-intl.** TR + EN at launch; `messages/{tr,en}.json`; runtime locale
  driven by user.locale (set via Mini App settings, defaults from Telegram language
  code on first /start).
- **No localStorage for state.** Frontend renders, backend owns. Telegram's
  DeviceStorage API exists but creates the same multi-device sync problem as
  localStorage — avoid for Phase 1.

### For vibe-designer

- **Telegram-native theme adaptation is mandatory.** Mini App MUST honor
  `Telegram.WebApp.themeParams` — palette adapts to user's Telegram dark/light setting.
  This is what distinguishes "feels like Telegram" from "looks like a webview website."
- **Single accent color, not a palette.** `#00D9C0` (electric teal) for buttons, active
  states, brand. Everything else inherits from Telegram theme variables. Reduces visual
  noise + ensures contrast in both modes.
- **Inter for body, no display font.** Telegram uses SF Pro / Roboto by platform; layering
  a different display font fights the native feel. Inter (400/500/600/700) for everything;
  weight does the typographic work.
- **Density: balanced.** Not data-dense (Linear/Datadog) and not spacious (Stripe
  marketing). Mini App is mostly read-and-tap; balanced spacing keeps tap targets safe
  while showing 6-8 items per phone screen.
- **Anti-list (must include in design.md):** glassmorphism cards, gradient text, stock
  illustrations of teamwork, emoji-as-decoration, neon glows, "AI-themed" cosmic imagery.
  Telegram users are functional-first; aesthetics serve the task.
- **Dark mode required, not optional.** Mini App auto-respects Telegram theme; light-only
  is not even an option. Both palettes specced in design.md.
- **Bot avatar + assets:** circular logo on dark teal background; legible at 48×48px in
  Telegram chat list. Avoid thin strokes.

## 9. Open research questions

- **Telegram username changes.** A user invited via `@oldname` who later changes to
  `@newname` — invite token approach (used in design) handles this, but edge case worth
  testing in Phase 2.
- **EU/UK GDPR self-host implications.** Self-hosters become data controllers. README
  should include a clear "data flows" section for self-host operators. Not blocking
  Phase 1.
- **Telegram bot username squatting.** `@listbull_bot` may be taken; backup names noted
  in CLAUDE.md gotchas section.
