# Design — listgram

> `architecture.md`. Engineering integration: see CLAUDE.md "Engineering conventions".
> Stack Defaults Styling: Tailwind + shadcn/ui (see `~/.claude/CLAUDE.md`).

## Schema

**Minimal Clean** with Telegram-native theme adaptation.

The Mini App lives inside Telegram. Its core design constraint is
"feel like Telegram, not like a webview" — which means: inherit Telegram's theme
variables (`Telegram.WebApp.themeParams`), add a single brand accent, and let
typography and spacing do the rest.

Marketing landing (`prod.listbull.org`) uses the same palette but in a fixed
light-mode variant — visitors haven't authenticated; we can't read their Telegram
theme. The marketing surface is small (hero + features grid + footer) so the
single-mode rendering is intentional.

## Adjective stack

**Functional, calm, fast, native, trustworthy.**

- **Functional** — every visible element earns its place; no decoration for its own sake.
- **Calm** — quiet palette, generous-but-not-spacious density, no neon or motion drama.
- **Fast** — perceived performance via skeletons, optimistic UI, sub-100ms tap response.
- **Native** — Telegram-tinted; users should not feel they "switched apps."
- **Trustworthy** — boring-on-purpose. Open source projects holding personal data
  shouldn't look flashy; calm signals stability.

## Color palette

The Mini App reads Telegram theme variables at runtime; values below are the **fallback
palette** when variables are absent (e.g. on the marketing landing or a stripped-down
test environment).

--muted:             #F4F4F5;
--muted-foreground:  #707579;  /* Telegram hint_color */
--accent:            #00D9C0;  /* listgram brand teal */
--accent-foreground: #FFFFFF;
--primary:           #00D9C0;  /* same as accent — single brand color */
--primary-foreground:#FFFFFF;
--border:            #E1E4E8;
--input:             #E1E4E8;
--ring:              #00D9C0;
--destructive:       #E53935;
--destructive-foreground: #FFFFFF;

```css

--card-foreground:   #F5F5F5;
--muted:             #232E3C;

--warning:           #FFB840;

```

**Runtime theme adapter** (`src/lib/telegram/theme-adapter.ts`):
- Subscribe to `Telegram.WebApp.onEvent('themeChanged')`.
- Map `themeParams.bg_color` → `--background`, `text_color` → `--foreground`,
  `hint_color` → `--muted-foreground`, `secondary_bg_color` → `--card`/`--muted`,
  `link_color` → ignored (we use accent).
- Brand accent `#00D9C0` stays constant across themes; everything else inherits.
- Re-apply on theme change without page reload (CSS vars on `<html>` element).

### Marketing landing (`prod.listbull.org`)

Light mode only. Palette identical to "Light mode" above. No theme adapter on

### Bot avatar

Circular asset. Solid `#0A1419` background, brand teal `#00D9C0` checkmark + send-arrow

## Typography

- **Body + UI:** `Inter` (variable font; weights 400, 500, 600, 700)
- **Display:** `Inter` (same family — single-font system)
- **Mono:** `Geist Mono` (only for code blocks if README preview rendered in app)

**Why no display font:** Telegram itself uses platform native (SF on iOS, Roboto on
Android). Layering Inter as both body AND display gives visual consistency without
fighting the native feel. Weight does typographic work.

**Self-hosting via `next/font`** (no Google Fonts CDN at runtime, GDPR-friendly).

### Type scale (Tailwind tokens)

| Token | px | weight | use |
|-------|----|--------|-----|
| `text-xs` | 12 | 500 | meta, timestamps |
| `text-sm` | 14 | 400 | secondary copy, item meta |
| `text-base` | 16 | 400 | item text, body |
| `text-lg` | 18 | 500 | list names, section headers |
| `text-xl` | 20 | 600 | page titles |
| `text-2xl` | 24 | 700 | onboarding hero, marketing H2 |
| `text-4xl` | 36 | 700 | marketing hero only |

Line height: 1.5 for `text-base` and below; 1.25 for `text-lg` and above.

## Density

**Balanced.**

- Item row height: 56px on mobile, 48px on desktop Mini App.
- List padding: `p-4` (16px) on mobile, `p-6` (24px) on desktop.
- Tap targets: minimum 44×44px (iOS HIG / WCAG 2.1).
- Inter-section spacing: `space-y-6` (24px) on mobile, `space-y-8` (32px) on desktop.

Not data-dense (Linear). Not spacious (Stripe marketing). Mini App users tap and read;
balanced density gives 6-8 items per phone screen without crowding.

## Dark mode

**Required.** Mini App auto-respects Telegram theme via runtime adapter. No manual
toggle inside Mini App (Telegram's setting is the source of truth).

Marketing landing: light only.

## Component patterns

| Pattern | Implementation |
|---------|----------------|
| Buttons | shadcn variants: `default` (accent teal), `secondary` (muted-bg), `destructive`, `ghost` |
| Inputs | shadcn `<Input />` + react-hook-form |
| Forms | shadcn `<Form />` + react-hook-form + zod |
| Tables | none — list-of-rows pattern only (custom `<ItemRow />` component) |
| Modals | shadcn `<Sheet />` (bottom sheet on mobile) for share dialog, item edit, settings |
| Confirms | shadcn `<AlertDialog />` for destructive actions only |
| Toasts | shadcn `<Sonner />` with `position="top-center"` (mobile-friendly) |
| Empty states | shared `<EmptyState />`: lucide icon (24px) + title + 1-line description + CTA button |
| Loading | Skeleton primitive matching layout — never spinners except button-internal |
| Checkboxes | custom — circular, 24×24, accent fill on check; matches Apple Reminders feel |
| Drag handle | lucide `<GripVertical />`, visible on hover (desktop) / always (mobile editing mode) |
| Toggle (settings) | shadcn `<Switch />` |

### Telegram MainButton

Mini App uses `Telegram.WebApp.MainButton` for primary actions (Save, Add, Share).
Replaces in-page submit buttons on edit screens. Theme-aware automatically.

### Telegram BackButton

Hardware-equivalent back navigation; we wire `Telegram.WebApp.BackButton` to
Next.js `router.back()` on routes that aren't root.

## Iconography

- Library: `lucide-react`
- Stroke width: 2px
- Sizes: 16 (inline), 20 (default in toolbar/list), 24 (feature icon, empty state)
- Color: inherits `currentColor` (defaults to `--foreground`)

**Custom icons:** none. lucide covers everything we need.

## Motion

- Library: CSS transitions only (no framer-motion). Mini App weight matters.
- Timings:
  - Tap feedback: 100ms
  - State change (check toggle): 150ms
  - Sheet open/close: 250ms
  - Skeleton-to-content fade: 200ms
- Easing: `ease-out` enter, `ease-in` exit
- **Reduced motion:** respected via `@media (prefers-reduced-motion: reduce)` —
  all transitions drop to 0ms, sheet uses opacity-only.

## Accessibility baseline

- WCAG AA contrast on all text (verified across light + dark with brand teal
  `#00D9C0`: against `#FFFFFF` = 2.7:1 ❌ for body, ✅ for large UI; against `#0A1419`
  = 9.1:1 ✅ everywhere). **Therefore: teal is button-bg + icon color, not body text.**
  Body text always uses `--foreground` against `--background`.
- Visible focus ring: 2px solid `--ring` with 2px offset. Never removed.
- ARIA via shadcn primitives (don't strip).
- Keyboard navigation: full Tab order, `Enter` to activate, `Esc` to dismiss sheets.
- Color is never the only signal: completed items get strikethrough text + reduced
  opacity, not just a color shift.
- Screen reader: lists announced as `role="list"` with item count; items as `role="listitem"`
  with full text + state ("completed" prefix when done).
- Lighthouse a11y target: ≥95 (verified in Phase 4 verification gate).

## Anti-list

Visual cliches and AI-product tics this design must actively avoid:

- **Glassmorphism** / frosted-glass cards / blur-behind backgrounds
- **Gradient text** (especially purple-to-pink, "AI" gradients)
- **Stock illustrations** of teamwork, lightbulbs, abstract people, "AI brain"
- **Neon glows** on buttons or borders ("cyberpunk SaaS")
- **Emoji-as-decoration** in serious UI (emoji are okay AS list-icons via user choice;
  not as randomly-sprinkled accents in headers)
- **Cosmic / space imagery** ("AI is the future" stars-and-nebulae aesthetic)
- **Gradient backgrounds** on hero sections
- **Skeumorphic shadows** ("paper" cards with drop shadows everywhere)

This anti-list is the universal filter for Claude Design prompts in
`docs/design-prompts.md`.

## Marketing vs app surface

- **Marketing surface** (`prod.listbull.org` root, OSS landing page):
  - Bigger type, hero-led, generous whitespace
  - Density: spacious on hero, balanced on features grid
  - Light mode only (no theme adapter — visitors aren't authed)
  - Single CTA: "Open in Telegram" (deeplink `https://t.me/listgram_bot?start=marketing`)

- **App surface** (Mini App, `/app/...`):
  - Functional density: balanced
  - Theme: Telegram-adapted (runtime variables)
  - No marketing chrome
  - Telegram MainButton + BackButton wired

- **Shared tokens:** brand teal `#00D9C0`, Inter typeface, lucide icons,
  border-radius `8px`.
- **Layout split:** `(marketing)` and `(app)` route groups in Next.js (see CLAUDE.md
  "Engineering conventions / Folder structure").

## Patterns derived from research.md insights

- **Telegram-native theme adaptation is mandatory** (Section 8) → drove the runtime
  theme adapter pattern; entire palette inherits from Telegram, brand color is the
  single override.
- **Single accent color, not a palette** (Section 8) → drove single-color brand system
  (`#00D9C0` as both `--accent` and `--primary`).
- **Inter for body, no display font** (Section 8) → drove the single-font typography
  decision.
- **Density: balanced** (Section 8) → drove 56px mobile / 48px desktop item heights.
- **Anti-list filter against generic AI tics** (Section 8) → expanded into the 8-item
  anti-list above.
- **Dark mode required, not optional** (Section 8) → drove dual-palette spec; light-only
  is not even an option for Mini App.
- **Mobile-first management is awkward in inline keyboards** (Section 6 #5) → drove
  decision to put bulk-select / drag-reorder in Mini App, not bot.
