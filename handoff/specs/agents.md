## Execution mode

**Hybrid** — Phase 1 (Foundation) is single-session; Phases 2 through 5 use the
multi-agent roster below. Phase 1 sets shared types, schema, scaffolding — all things
that benefit from one consistent author. From Phase 2 onward, parallelizable work
across surfaces (bot vs Mini App vs LLM tooling) earns the multi-agent overhead.

## Agent roster

5 agents. Justifications:

1. **Architect-agent** — folder structure validation, Drizzle schema review, type
   freezing in `src/lib/types/`, design.md token mapping for UI bundle handoffs.
   Critical because listbull has two surfaces (bot + Mini App) sharing types and
   contracts; misalignment between them is the most likely failure mode.
2. **Backend-agent** — bot webhook handler, slash command implementations, LLM tool
   implementations, Drizzle queries, migrations, cron job, activity_log writes,
   webhook secret + initData verification. Owns "the server."
3. **Frontend-agent** — Mini App routes, components, optimistic UI, Telegram theme
   adapter, share UI, settings UI, accessibility. Owns "the visible surface."
4. **AI-agent** — LLM system prompt, tool schema definitions (zod), conversation
   history slicing, multi-turn flow, prompt-versioning. Specialized because the LLM
   surface is the wedge feature; one owner avoids prompt drift across phases.
5. **Reviewer-agent** — per-phase lint/tsc runs, security pass on initData/webhook
   secret/encryption-at-rest, OSS-quality pass (README, LICENSE, .env.example, Docker
   Compose validation), Lighthouse a11y in Phase 4.

No DB-agent: schema is 7 tables, Backend-agent owns migrations directly. No
DevOps-agent: Dokploy + GitHub Actions are well-trodden in this stack; CI is a copy
from the SocialCal config. No Marketing-agent: only one landing page, Frontend-agent
ships it during Phase 4.

## Inter-agent contracts

### Architect ↔ everyone

- **Shared types live at**: `src/lib/types/` (Architect-owned, frozen after Phase 1)
- **Schema source of truth**: `src/lib/db/schema.ts` (Architect-owned; Backend reads)
- **New shared type workflow**:
  1. Architect adds the type, exports from `src/lib/types/index.ts`
  2. Architect commits with message `types: <change>`
  3. Other agents import; never declare equivalents elsewhere

### Backend ↔ Frontend

- **API contract format**: zod schemas in `src/lib/validators/`. Both sides import.
- **Type inference**: validators export `type X = z.infer<typeof xSchema>` for shared
  request/response types; both sides use the inferred types.
- **New endpoint workflow**:
  1. Backend writes route + zod schema in `src/lib/validators/<feature>.ts`
  2. Backend exports the inferred type
  3. Backend ships the route, ensures `npm run build` passes
  4. Frontend imports type, uses it for fetch calls + form
- **Error envelope**: `{ ok: true, data }` / `{ ok: false, error: { code, message } }` —
  consistent across all routes
- **Polling sync (Phase 1+)**: shared list views poll every 5s with TanStack Query.
  Backend's `GET /api/lists/:id/items` returns fresh state; client reconciles. No
  websockets in Phase 1.

### Backend ↔ AI

- **Tool schema source of truth**: `src/lib/ai/tools.ts` (AI-agent owns).
  Each tool has: zod input schema + zod output schema + LLM-facing description.
- **Tool execution**: lives in `src/lib/server/tools/<tool-name>.ts` (Backend-agent
  owns). Each executor:
  1. Receives validated input (zod-parsed against tool's input schema)
  2. Runs business logic in a single Drizzle transaction
  3. Writes activity_log row
  4. Returns output matching tool's output schema
- **Bot webhook flow**:
  1. Webhook handler (Backend) receives Update, looks up user, loads conversation
     history (last 30 messages)
  2. Backend calls AI client (`src/lib/ai/respond.ts`) with: system prompt +
     conversation history + new message + tool schemas
  3. LLM returns: assistant message OR tool calls
  4. Backend executes tool calls (transactionally) → returns results to AI
  5. AI does follow-up turn if needed; returns final assistant message
  6. Backend persists messages to DB + sends Telegram reply
- **Conversation history handoff**: AI-agent owns `src/lib/ai/conversation.ts`
  (the slicing rules). Backend imports and uses it.

### Frontend ↔ Telegram WebApp SDK

- **Init**: Mini App `(app)` layout calls `WebApp.ready()` on mount, mounts theme
  adapter, sets up MainButton/BackButton lifecycle.
- **Auth**: initData query string is forwarded to Backend's `POST /api/auth/telegram`,
  which verifies HMAC and issues session cookie. Frontend receives cookie via
  Better Auth client; subsequent fetches authenticated automatically.
- **Theme**: `src/components/telegram/theme-provider.tsx` reads `WebApp.themeParams`,
  applies CSS variables to `<html>` element, subscribes to `themeChanged` event.

### Reviewer ↔ everyone

- **Review trigger**: at end of each phase, orchestrator invokes Reviewer with
  the phase number + scope summary.
- **Review checklist** (Reviewer-agent runs):
  1. `npm run lint` clean
  2. `npx tsc --noEmit` clean
  3. (Phase 4) `npm run test` clean
  4. (Phase 4) `npm run e2e` clean
  5. (Phase 4) Lighthouse a11y on Mini App ≥95
  6. (Phase 4) `docker-compose up` works on clean machine
  7. Security pass: search for hardcoded secrets, unverified initData, missing
     webhook secret check, plaintext API key storage
  8. OSS pass (Phase 4): README, LICENSE, .env.example, CONTRIBUTING all present
- **Output**: `docs/review-phase-<N>.md` with pass/fail + remediation list. If fail,
  orchestrator routes back to relevant agent for fix; Reviewer re-runs.

## Conflict resolution

When two agents touch overlapping files:

1. **Folder ownership rule**: the agent listed as `Owns` for the path wins by default.
2. **Cross-cutting type changes** (`src/lib/types/`): Architect-agent has final say.
   If Backend or AI need a new type, request it via PR; Architect adds it; everyone
   else imports.
3. **Validators** (`src/lib/validators/`): Backend-agent owns. Frontend imports.
4. **Tool schemas vs tool executors** (the most likely friction):
   - Schema in `src/lib/ai/tools.ts` is AI-agent's.
   - Executor in `src/lib/server/tools/<name>.ts` is Backend-agent's.
   - If they disagree, AI-agent's schema wins (LLM-facing contract is the immovable
     surface); Backend adapts the executor.
5. **Migrations** (`src/lib/db/migrations/`): Backend owns. Architect reviews schema
   diff before Backend runs `drizzle-kit generate`.
6. **shadcn primitives** (`src/components/ui/`): Frontend regenerates via `shadcn add`;
   never edit manually.
7. **README, LICENSE, .env.example, docker-compose.yml**: Reviewer-agent owns
   (Phase 4); Backend reviews docker-compose.

If conflict cannot be resolved by the rules above, escalate to the Claude Code
orchestrator session (i.e. the user). Don't merge contested code.
