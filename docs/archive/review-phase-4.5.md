## Code-review notes

### Strengths

1. **Atomic transactions**: every executor that touches multiple tables uses `db.transaction()` (Inv-1). Verified in: workspace-pivot.ts (4 inserts per user), update-workspace.ts (lists + activity_log), remove-workspace-member.ts (cascade to items + list_members + activity_log).
2. **Workspace gate consistency**: `resolveList` + `userCanWriteList` + `userCanReadList` all join `lists.workspace_id = ctx.workspaceId`. No remaining unscoped queries surfaced via grep.
3. **Type safety preserved**: zero `any` in new code; `as unknown as` cast only at the Stripe API-version boundary (SDK enum lag).
4. **Migration reversibility**: schema-only steps (1-2) reversible by dropping new tables/columns; data-mutation steps (3-9) recoverable via hourly Postgres backup.
5. **Idempotent migration**: workspace-pivot.ts checks for existing rows before inserting (default bot, Personal Workspaces, workspace_bots, bot_users). Re-running = no-op.
6. **Token encryption**: default bot's Telegram token encrypted via AES-256-GCM (same envelope as BYOK keys); never logged.

### Risks deferred to Phase 5

1. **In-memory idempotency** (src/lib/billing/idempotency.ts): single-pod safe. If listbull scales beyond one Dokploy container, two pods could process the same Stripe event. Phase 5 swap to Upstash KV mandatory before BILLING_ENFORCE=true production deploy.
2. **Today view tz approximation** (src/app/(app)/views/today/page.tsx): EOD calculation uses Intl.DateTimeFormat to extract the user's local Y/M/D, then constructs a JS Date with system-local time. Acceptable when bot host runs UTC (current default); inaccurate by ±1 day if bot host changes timezone. Phase 5: replace with date-fns-tz or a Postgres-side `at time zone` formula.
3. **Iyzico full checkout flow**: Phase 4.5 only verifies signatures + logs events. Real subscription upserts mirror Stripe handler shape — Phase 5 deliverable.
4. **Tier-enforce never blocks**: BILLING_ENFORCE defaults to "false". Phase 5 flip + tier-check edge-case audit (concurrent member-invite race against limit) before SaaS launch.
5. **Tag vocabulary cap query** (set-item-attributes.ts): builds raw SQL with `sql.raw`. Inputs are workspace_id UUIDs sourced from a Drizzle query (not user input), so injection-safe — but the raw SQL pattern is unusual; Phase 5 audit should verify the safety claim.

### Test coverage gaps

- No unit tests for the 6 new executors yet. The existing executor test fixture (executors-input-validation.test.ts) was extended with workspaceId. Phase 5 should add per-executor happy-path + workspace-isolation tests (cross-workspace access denied, ambiguous_workspace surfacing, etc.).
- No webhook handler tests. Phase 5 + Stripe CLI signature tooling makes these tractable.
