## What's new

Phase 4.5 lands six concerns in one pass:

1. **Workspace layer** (`workspaces`, `workspace_members`,
   `users.active_workspace_id`, `lists.workspace_id`). Every existing
   user gets an auto-Personal Workspace; every existing list backfills
   to that user's Personal Workspace.
2. **Item discipline** (`items.status`, `items.priority`, `items.tags`).
   Additive columns with defaults so existing rows remain valid.
3. **Billing infrastructure** (`subscriptions`, `billing_customers`).
   Stripe + Iyzico providers; webhook handlers + tier middleware
   wired but **enforcement OFF** (logs only). Phase 5 flips
   enforcement to active.
4. **Multi-bot schema seed** (`bots`, `workspace_bots`, `bot_users`).
   The default platform bot becomes a single seeded row in `bots`;
   every existing user/workspace binds to it. **No white-label
   registration UI in Phase 4.5** — that lands in Phase 5. Schema
   exists now so Phase 5 doesn't trigger a second migration.
5. **6 new LLM tools** (`switch_workspace`, `list_workspaces`,
   `invite_to_workspace`, `remove_workspace_member`,
   `update_workspace`, `set_item_attributes`). All 18 existing tools
   are retrofitted to scope by the active workspace.
6. **License-verify middleware stub** — env-gated JWT signature check,
   default DISABLED. Issuance + admin UI defer to Phase 6.

## Schema deltas — full specification

### NEW table `workspaces`

```ts
{
  id:               uuid pk
  name:             text not null
  slug:             text unique not null  -- URL-safe, auto-generated
  tier:             text not null default 'free'  -- 'free' | 'team' | 'workspace'
  isPersonal:       boolean not null default false
  ownerId:          uuid not null fk users.id
  memberLimit:      integer not null  -- cached from tier (1/5/15)
  archivedAt:       timestamptz nullable
  createdAt:        timestamptz not null defaultNow
  updatedAt:        timestamptz not null defaultNow
}
```

**Indexes:**
- `workspaces_personal_per_owner_uq`: unique partial on `(owner_id)` where `is_personal = true` — exactly one Personal Workspace per user
- `workspaces_owner_idx`: btree on `(owner_id)`
- `workspaces_slug_idx`: unique btree on `(slug)`

**Notes:**
- `memberLimit` denormalized from tier for fast checks. Refreshed by
  Billing-agent's webhook handler on subscription state changes
  (atomic update inside webhook idempotency block).
- `slug` is auto-generated from `name` (lower, kebab-case, deduped with
  numeric suffix on collision). Mutable: rename → re-slug. Old slug
  doesn't redirect (workspaces are private; no SEO concern).

### NEW table `workspace_members`

```ts
{
  id:          uuid pk
  workspaceId: uuid not null fk workspaces.id on delete cascade
  userId:      uuid not null fk users.id on delete cascade
  role:        text not null  -- 'owner' | 'admin' | 'editor' | 'viewer' | 'guest'
  invitedBy:   uuid nullable fk users.id
  acceptedAt:  timestamptz not null defaultNow
  createdAt:   timestamptz not null defaultNow
  updatedAt:   timestamptz not null defaultNow
}
```

**Indexes:**
- `workspace_members_workspace_user_uq`: unique on `(workspace_id, user_id)`
- `workspace_members_user_idx`: btree on `(user_id)` — lookup "all my workspaces"

**Role semantics:**
- `owner`: full control + billing + delete
- `admin` (Workspace tier only): everything owner can do EXCEPT
  delete workspace + transfer ownership + change billing
- `editor`: mutate items + create lists; cannot mutate members or
  workspace settings
- `viewer` (Team+ tier): read-only across workspace
- `guest` (Team+ tier): single-list access only — workspace_members
  row + a list-scoped list_members row jointly grant access. Phase
  4.5 ships the schema; the UI for guest invites lands Phase 5+.

### NEW table `subscriptions`

```ts
{
  id:                       uuid pk
  workspaceId:              uuid unique not null fk workspaces.id on delete cascade
  provider:                 text not null  -- 'stripe' | 'iyzico' | 'manual'
  providerCustomerId:       text not null
  providerSubscriptionId:   text not null
  tier:                     text not null  -- 'free' | 'team' | 'workspace'
  status:                   text not null  -- 'active' | 'past_due' | 'canceled' | 'trialing'
  currentPeriodStart:       timestamptz nullable
  currentPeriodEnd:         timestamptz nullable
  cancelAtPeriodEnd:        boolean not null default false
  createdAt:                timestamptz not null defaultNow
  updatedAt:                timestamptz not null defaultNow
}
```

**Indexes:**
- `subscriptions_workspace_id_idx`: unique on `(workspace_id)`
- `subscriptions_provider_sub_idx`: unique on `(provider, provider_subscription_id)` — webhook lookup

**Free-tier note:** Free workspaces do NOT have a `subscriptions` row.
Tier middleware treats absence-of-row as Free. This keeps the table
small (only paying customers populate it) and avoids a webhook needing
to backfill on signup.

### NEW table `billing_customers`

```ts
{
  id:                  uuid pk
  userId:              uuid not null fk users.id
  provider:            text not null  -- 'stripe' | 'iyzico'
  providerCustomerId:  text not null
  email:               text not null
  country:             text not null  -- ISO 3166-1 alpha-2 (TR, DE, US...)
  taxId:               text nullable  -- TR vergi no or EU VAT ID
  createdAt:           timestamptz not null defaultNow
  updatedAt:           timestamptz not null defaultNow
}
```

**Indexes:**
- `billing_customers_user_provider_uq`: unique on `(user_id, provider)`
- `billing_customers_provider_customer_uq`: unique on `(provider, provider_customer_id)`

**Notes:**
- One billing customer record per (user, provider). A user cannot
  have both a Stripe and Iyzico subscription simultaneously — country
  determined at first paid signup locks provider.
- `taxId` collected at checkout for KDV-compliant invoicing (TR) or
  EU VAT (Stripe Tax).

### NEW table `bots`

```ts
{
  id:                          uuid pk
  telegramBotId:               bigint unique not null  -- Telegram's numeric bot ID
  telegramBotUsername:         text unique not null    -- '@listbull_bot' (without @)
  telegramBotTokenEncrypted:   text not null           -- AES-256-GCM, same helper as BYOK
  webhookSecret:               text not null           -- per-bot Telegram webhook secret
  isDefault:                   boolean not null default false  -- exactly one row with true
  createdBy:                   uuid nullable fk users.id  -- null for default platform bot
  createdAt:                   timestamptz not null defaultNow
  updatedAt:                   timestamptz not null defaultNow
}
```

**Indexes:**
- `bots_telegram_id_uq`: unique on `(telegram_bot_id)`
- `bots_username_uq`: unique on `(telegram_bot_username)`
- `bots_default_uq`: unique partial on `(is_default)` where `is_default = true` — exactly one default bot

**Phase 4.5 seed:** migration script inserts ONE row representing the
existing `@listbull_bot` (or `TELEGRAM_BOT_USERNAME` from env), with
`is_default = true`, `created_by = NULL`. Token encrypted from
`TELEGRAM_BOT_TOKEN`; webhook secret from `TELEGRAM_WEBHOOK_SECRET`.

**Phase 4.5 webhook routing:** stays single-route at
`/api/bot/webhook` (existing). Implementation reads the only `bots`
row where `is_default = true` and uses its token. Phase 5 introduces
per-bot routing at `/api/bot/webhook/[bot_id]`.

### NEW junction table `workspace_bots`

```ts
{
  id:           uuid pk
  workspaceId:  uuid not null fk workspaces.id on delete cascade
  botId:        uuid not null fk bots.id on delete cascade
  isPrimary:    boolean not null default false  -- workspace's white-label bot
  createdAt:    timestamptz not null defaultNow
  updatedAt:    timestamptz not null defaultNow
}
```

**Indexes:**
- `workspace_bots_pair_uq`: unique on `(workspace_id, bot_id)`
- `workspace_bots_workspace_idx`: btree on `(workspace_id)`
- `workspace_bots_bot_idx`: btree on `(bot_id)`

**Cardinality:** many-to-many. A workspace may be reachable via the
default bot AND a white-label bot during transition (Phase 5
admin chooses to add or replace). `is_primary = true` means "this is
the workspace's own bot"; default platform bot rows always have
`is_primary = false`.

**Phase 4.5 seed:** migration inserts one row per (existing workspace,
default bot) pair with `is_primary = false`.

### NEW table `bot_users`

```ts
{
  botId:      uuid not null fk bots.id on delete cascade
  userId:     uuid not null fk users.id on delete cascade
  startedAt:  timestamptz not null defaultNow  -- when the user /start'ed this bot
  createdAt:  timestamptz not null defaultNow
}
```

**Primary key:** composite `(bot_id, user_id)` — no surrogate id.

**Indexes:**
- `bot_users_user_idx`: btree on `(user_id)` — "which bots can this user receive DMs from"
- `bot_users_bot_idx`: btree on `(bot_id)`

**Use:** before `share_list` invite or reminder dispatch sends a DM,
verify a `bot_users` row exists for `(bot_id, user_id)`. If missing,
DM attempt will 403 (Telegram requires the user to have started the
bot). The `share_list` flow already handles this with
`invitee_dm_failed`; reminder dispatch (Phase 5) falls back to default
platform bot when the workspace bot has no row.

**Phase 4.5 seed:** migration inserts one row per existing user for
the default platform bot — they all `/start`ed it once to enter
listbull. `started_at = users.created_at` (best-effort backfill).

### MODIFIED table `users`

Add column:
- `activeWorkspaceId: uuid nullable fk workspaces.id` — user's currently-selected workspace. Nullable because at user creation time the Personal Workspace doesn't exist yet (resolved in the same migration via UPDATE; runtime new-user signup also writes both rows in one transaction).

**Resolution rule (Phase 4.5):**
- Mini App + bot read `users.active_workspace_id` directly.
- Phase 5 adds bot-aware overlay: if incoming bot is workspace-bound
  (`workspace_bots.is_primary = true`), runtime context = that
  workspace, regardless of `active_workspace_id`. The DB column stays
  the user's "default fallback" preference.

### MODIFIED table `lists`

Add column:
- `workspaceId: uuid not null fk workspaces.id on delete cascade`

The `ownerId` column STAYS — represents the user who created the
list (informational + audit). Workspace membership controls access
going forward, not list ownership. `lists_owner_inbox_unique` partial
unique index (one Inbox per owner) is REPLACED with
`lists_workspace_inbox_unique` (one Inbox per workspace) — the Inbox
is now a workspace-level concept, not a user-level one. (Personal
Workspace gets its Inbox; shared workspaces also have their own
Inbox if any free-form items land there.)

Add index:
- `lists_workspace_id_idx`: btree on `(workspace_id, archived_at)`

### MODIFIED table `items`

Add columns:
- `status: text not null default 'open'` — `'open' | 'in_progress' | 'blocked' | 'done'`
- `priority: text not null default 'normal'` — `'low' | 'normal' | 'high'`
- `tags: text[] not null default '{}'`

**Indexes:**
- `items_status_idx`: btree on `(list_id, status)`
- `items_tags_gin`: GIN on `(tags)`

**`is_done` retention:** the existing `is_done` column STAYS for
backward compatibility through Phase 4.5. Treated as derived from
`status` going forward (`is_done = (status = 'done')`); writes touch
both columns. Phase 5+ decision whether to drop `is_done` (saves
1 byte per row but requires touching every existing executor).
**Migration backfill:** `status = 'done'` if `is_done = true`,
`status = 'open'` otherwise.

## Migration runbook (single transactional script)

The migration is **one transactional Drizzle migration**
(`drizzle/0001_workspace_billing_bot.sql`) plus **one TypeScript data
migration** (`src/lib/server/migrations/workspace-pivot.ts`) run after
the SQL migration applies.

### Step-by-step

1. **Add new tables** (additive, no data touched):
   `workspaces`, `workspace_members`, `subscriptions`,
   `billing_customers`, `bots`, `workspace_bots`, `bot_users`. All
   indexes on these tables are created here.

2. **Add nullable columns** to existing tables:
   - `users.active_workspace_id` (nullable)
   - `lists.workspace_id` (nullable — temporarily)
   - `items.status`, `items.priority`, `items.tags` (with defaults — existing rows auto-populate)

3. **Backfill `status`** from `is_done`:
   ```sql
   UPDATE items SET status = 'done' WHERE is_done = true;
   -- 'open' is the default for the rest
   ```

4. **Seed default bot:**
   ```sql
   INSERT INTO bots (telegram_bot_id, telegram_bot_username, telegram_bot_token_encrypted, webhook_secret, is_default)
   VALUES (env_TELEGRAM_BOT_ID, env_TELEGRAM_BOT_USERNAME, encrypt(env_TELEGRAM_BOT_TOKEN), env_TELEGRAM_WEBHOOK_SECRET, true);
   ```
   Run from the TS migration script (`workspace-pivot.ts`) so the
   encryption helper has access to env. Drizzle SQL migration cannot
   call app-layer encryption.

5. **Seed Personal Workspace per existing user:**
   ```sql
   -- Pseudocode; actual TS:
   FOR each user IN (SELECT id FROM users):
     workspace_id := INSERT INTO workspaces (name, slug, tier, is_personal, owner_id, member_limit)
                     VALUES ('Personal', user.id || '-personal', 'free', true, user.id, 1)
                     RETURNING id;
     INSERT INTO workspace_members (workspace_id, user_id, role)
            VALUES (workspace_id, user.id, 'owner');
     UPDATE users SET active_workspace_id = workspace_id WHERE id = user.id;
   ```

6. **Backfill `lists.workspace_id`:**
   ```sql
   UPDATE lists
   SET workspace_id = (
     SELECT id FROM workspaces
     WHERE owner_id = lists.owner_id AND is_personal = true
   );
   ```

7. **Make `lists.workspace_id` NOT NULL:**
   ```sql
   ALTER TABLE lists ALTER COLUMN workspace_id SET NOT NULL;
   ```

8. **Drop old `lists_owner_inbox_unique` partial index; add
   `lists_workspace_inbox_unique`:**
   ```sql
   DROP INDEX lists_owner_inbox_unique;
   CREATE UNIQUE INDEX lists_workspace_inbox_unique
     ON lists (workspace_id) WHERE is_inbox = true;
   ```

9. **Seed `workspace_bots` + `bot_users` for default bot:**
   ```sql
   INSERT INTO workspace_bots (workspace_id, bot_id, is_primary)
     SELECT w.id, b.id, false FROM workspaces w, bots b WHERE b.is_default = true;
   INSERT INTO bot_users (bot_id, user_id, started_at)
     SELECT b.id, u.id, u.created_at FROM bots b, users u WHERE b.is_default = true;
   ```

### Reversibility

- **Steps 1-2** fully reversible: drop the new tables; drop the new
  columns. Existing app code keeps working (it doesn't reference
  workspace_id yet at this point).
- **Steps 3-6** reversible: re-derive `status` from `is_done` (or
  ignore the new column); delete all `workspaces` + `workspace_members`
  rows; null out `lists.workspace_id` (still nullable at this point).
- **Step 7 (NOT NULL on lists.workspace_id)** is the destructive
  threshold. After this step, rolling back requires either restoring
  from backup OR manually re-deleting the workspace_id column (which
  loses workspace assignment).
- **Steps 8-9** mechanically reversible.

**Rollback policy:** if the migration fails between steps 1 and 6,
abort transaction and the DB is unchanged. If steps 7+ have applied
and rollback is needed, restore from the pre-migration backup. The
hourly Postgres backup pipeline (`hetzner-postgres-backup.py` per
global CLAUDE.md) provides the recovery point.

### Verification post-migration

Reviewer-agent runs these queries against the post-migration test DB
snapshot before approving prod cutover:

```sql
-- Every user has exactly one Personal Workspace
SELECT user_id, COUNT(*) FROM (
  SELECT u.id AS user_id FROM users u
  LEFT JOIN workspaces w ON w.owner_id = u.id AND w.is_personal = true
  WHERE w.id IS NULL
) GROUP BY user_id;
-- Expected: 0 rows

-- Every list has a workspace_id
SELECT COUNT(*) FROM lists WHERE workspace_id IS NULL;
-- Expected: 0

-- Every workspace has its owner as a member
SELECT w.id FROM workspaces w
LEFT JOIN workspace_members m ON m.workspace_id = w.id AND m.user_id = w.owner_id AND m.role = 'owner'
WHERE m.id IS NULL;
-- Expected: 0 rows

-- Default bot seeded exactly once
SELECT COUNT(*) FROM bots WHERE is_default = true;
-- Expected: 1

-- Every user can DM the default bot
SELECT COUNT(*) FROM users u
LEFT JOIN bot_users bu ON bu.user_id = u.id
LEFT JOIN bots b ON b.id = bu.bot_id AND b.is_default = true
WHERE bu.user_id IS NULL;
-- Expected: 0

-- Every workspace bound to default bot
SELECT COUNT(*) FROM workspaces w
LEFT JOIN workspace_bots wb ON wb.workspace_id = w.id
LEFT JOIN bots b ON b.id = wb.bot_id AND b.is_default = true
WHERE wb.bot_id IS NULL;
-- Expected: 0

-- items.status backfill correct
SELECT COUNT(*) FROM items WHERE (is_done = true) <> (status = 'done');
-- Expected: 0
```

## Workspace context resolution contract

Every authenticated request operates in the context of one workspace.
Resolution order — implemented as
`src/lib/server/middleware/workspace-context.ts` (Backend-owned):

1. **Mini App `?workspace=<slug>` query param** — explicit override,
   highest priority. Persisted to cookie on switch.
2. **Cookie `lb_active_workspace_id`** — set by the workspace switcher
   UI; second priority for Mini App routes.
3. **`users.active_workspace_id`** — DB-stored default. Used by bot
   handlers (no cookie context) and Mini App when no explicit
   override.
4. **Fallback: user's Personal Workspace** — derived via
   `(SELECT id FROM workspaces WHERE owner_id = ? AND is_personal = true)`.
   Should never be needed at runtime (every user has a Personal
   Workspace post-migration), but kept as defense-in-depth.

**Phase 5 overlay (NOT in 4.5):** when an incoming Telegram update
arrives via a workspace-bound bot (i.e., `workspace_bots.is_primary =
true` for that bot_id), runtime context overrides steps 1-4 with that
bot's bound workspace. The DB column stays the user's preference;
runtime resolution is bot-aware.

**Membership gate:** middleware MUST verify the resolved workspace_id
has a matching `workspace_members` row for the requesting user.
Mismatch → 403 forbidden. Prevents URL-fuzzing access to other users'
workspaces.

## Tier enforcement contract

`src/lib/server/middleware/tier-enforce.ts` is the single point where
tier limits are checked. Phase 4.5 ships the middleware in **logs-only
mode** (env `BILLING_ENFORCE=false` default). Phase 5 flips to
`BILLING_ENFORCE=true`.

### `checkTier` helper signature

Owned by Billing-agent at `src/lib/billing/tier-check.ts`:

```ts
type TierCheckAction =
  | { type: 'create_workspace' }
  | { type: 'invite_member'; current_count: number }
  | { type: 'create_list' }
  | { type: 'set_org_api_key' }
  | { type: 'use_custom_role' };

type TierCheckResult =
  | { ok: true }
  | {
      ok: false;
      reason: 'tier_exceeded' | 'past_due_locked' | 'workspace_count_exceeded';
      currentTier: 'free' | 'team' | 'workspace';
      upgradeTo: 'team' | 'workspace';
      message: string;  // user-facing copy (TR/EN, locale-aware caller)
    };

export async function checkTier(
  workspaceId: string,
  action: TierCheckAction,
): Promise<TierCheckResult>;
```

Backend's `tier-enforce.ts` middleware calls this on every mutation
route. In Phase 4.5 (`BILLING_ENFORCE=false`), middleware:

1. Calls `checkTier`.
2. **Logs** result with action + workspace_id + outcome.
3. **Always proceeds** regardless of result.

In Phase 5 (`BILLING_ENFORCE=true`), step 3 changes to: if `ok: false`,
return 402 + body `{ error: { code: result.reason, message:
result.message, upgradeTo: result.upgradeTo } }`.

### Tier limits (cached on `workspaces.member_limit`)

| Tier | members | workspaces/user | custom roles | org API key | activity log retention |
|---|---|---|---|---|---|
| Free | 1 | 1 (Personal only) | — | — | 30 days |
| Team | 5 | 1 | — | — | 90 days |
| Workspace | 15 | 3 | ✓ | ✓ | unlimited |

`memberLimit` cached on the workspaces row to avoid a join on every
mutation. Refreshed inside Billing-agent's webhook handler in the same
transaction that updates `subscriptions.tier`.

## License-verify middleware stub

`src/lib/server/middleware/license-verify.ts` is wired into the
request pipeline but **default DISABLED** via env:

```env
LICENSE_VERIFY_ENABLED=false  # default
LICENSE_PUBLIC_KEY=             # PEM string, set on self-host
```

When `LICENSE_VERIFY_ENABLED=false`: middleware is a no-op pass-through.

When `true`: middleware reads `LICENSE_KEY` from env, verifies JWT
signature against `LICENSE_PUBLIC_KEY` (Ed25519), checks
`exp` + `revoked_at` (offline list), and gates Team/Workspace tier
features. Free-tier features always pass.

**Phase 4.5 ships the middleware skeleton + signature-verification
helper. Phase 6 ships:**
- License issuance endpoint (`/api/admin/license/issue` — Billing-agent)
- Admin dashboard (Backend + Frontend)
- Revocation list management
- Self-host operator workflow

The skeleton means Phase 6 is just "fill in issuance"; no schema
changes Phase 6.

## Architectural decisions made on day 0

1. **`workspace_bots` cardinality: many-to-many.** Phase 5's white-
   label flow ADDS a second binding (workspace bound to default + own
   bot during transition). Admin can later set the white-label bot as
   primary or revoke the default binding explicitly. Rationale:
   graceful migration, no surprise loss-of-access.

2. **Bot instance pool: deferred to Phase 5.** Phase 4.5 schema-only;
   single grammY instance keeps running against the default bot. Phase
   5 architect pass decides single-router vs instance-per-bot based on
   memory profile.

3. **License key payload binding: per-Workspace-tier subscription.**
   One license unlocks the 3-workspace allotment; workspaces array in
   payload. Rationale: matches Workspace tier's "up to 3 workspaces"
   semantic without forcing the operator to manage 3 separate keys.

4. **Migration ordering: single transactional Drizzle migration.**
   All schema changes apply atomically. The TS data migration
   (workspace-pivot.ts) runs immediately after, also wrapped in a
   transaction. Two-step (SQL DDL + TS DML) instead of one because
   Drizzle migrations can't call app-layer encryption (token seeding
   needs `encrypt()`).

5. **`is_done` column retention.** Kept through Phase 4.5 for backward
   compat (every existing executor reads it). Dual-write pattern:
   writes touch both `status` and `is_done`. Decision to drop deferred
   to Phase 5 with full executor audit.

6. **Activity log scope: list-level stays, no workspace-level events
   yet.** Phase 4.5 doesn't add `workspace_created` /
   `workspace_member_added` activity actions. Workspace events
   (membership changes, tier changes, billing) belong to a separate
   audit stream Phase 5+ may introduce. For Phase 4.5, mutations on
   the workspace shell (create, rename, member add) just write a
   single `activity_log` row with `entity_type = 'workspace'` (NEW
   value) — extending the enum, not adding actions.

7. **Tag vocabulary: workspace-scoped, free-form.** No `tags` table;
   `items.tags` is a text array. First introduction of a tag is just
   "type the chip name." Workspace settings page (Phase 4.5 Frontend)
   shows the union-of-tags-currently-in-use as autocomplete source.
   Cap at 20 unique tags per workspace enforced in `set_item_attributes`
   executor (count-distinct query at write time; reject 21st).

8. **`activity_log.entity_type` enum extended:** add `'workspace'` to
   the existing `'item' | 'list' | 'member'`. Action enum extended:
   `workspace_created`, `workspace_renamed`, `workspace_member_added`,
   `workspace_member_removed`, `workspace_member_role_changed`. The
   Phase 1 frozen `ActivityAction` type in `src/lib/types/index.ts`
   gets these 5 additions in this pass — first extension since freeze,
   justified by feature scope.

## Risks + mitigations

1. **Migration on production data**: highest risk. Mitigated by
   single transactional script + Reviewer-agent dry-run on prod-DB
   snapshot + hourly backup as recovery point.
2. **Existing user UX disruption**: Mini App suddenly shows "Personal
   Workspace" header. Mitigated by Frontend's first-load tooltip
   (Phase 4.5 deliverable, A3 scope).
3. **Tier-middleware drift**: logs-only mode in Phase 4.5 means
   bugs in `checkTier` go unnoticed until Phase 5 enforce flip.
   Mitigated by structured logging (action + decision + reason)
   that Phase 5 audit replays before flipping enforce.
4. **Bot token encryption helper availability**: TS migration script
   needs `encrypt()` from `@/lib/auth/crypto` (used for BYOK keys).
   Backend-agent confirms this helper is server-only and importable
   from a CLI migration runner. Document the runner command:
   `npx tsx src/lib/server/migrations/workspace-pivot.ts`.
5. **`activity_log.entity_type` enum drift**: app-layer enum
   (TypeScript) vs DB column (text). Existing pattern is "no DB
   constraint, app validates." We continue the pattern; reviewer
   pass adds a defensive Vitest case ensuring `'workspace'` round-
   trips correctly through the activity log.
