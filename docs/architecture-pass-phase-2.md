# Architecture Pass — Phase 2 (Core Conversational + Manage)

> Generated by Architect-agent on 2026-05-01.
> This is the canonical contract for AI-agent (`src/lib/ai/**`) and
> Backend-agent (`src/lib/server/**`) for Phase 2. Both agents read
> this file as their starting point. Any deviation = Phase 2 verification
> failure; deviations require new Architect invocation, not in-flight
> hot-fixes.

## Scope

Phase 2 ships the LLM tool layer — six tools that let the bot mutate
items on behalf of the user via natural language — plus conversation
history persistence and BYOK encryption. Sharing, reminders,
assignment are deferred to Phase 3; they appear here only as referenced
types (e.g. `assignee_id` is read but not written by Phase 2 tools).

Six Phase-2 tools:

1. `create_item`
2. `search_items`
3. `update_item`
4. `complete_item`
5. `delete_item`
6. `list_lists`

Three tools deferred to Phase 3 (NOT to be implemented in Phase 2):
`share_list`, `schedule_reminder`, `assign_item`.

---

## Shared types (added to `src/lib/types/index.ts` this phase)

| Type | Purpose |
|---|---|
| `ToolCall` | One assistant tool invocation: `{ id, name, input: unknown }`. Mirror of `messages.tool_calls` JSONB element. |
| `ToolResult` | One executor result: `{ toolCallId, output: unknown }`. Persisted as a `role='tool'` message row. |
| `ConversationMessage` | Discriminated union over `MessageRole` for in-memory LLM orchestration. NOT a DB row. |
| `MessageWithToolCalls` | DB row with JSONB `tool_calls` parsed to `ToolCall[] \| null`. What query helpers should return. |
| `ItemSnapshot` | JSON-safe shape stored in `activity_log.payload_before` / `payload_after` when `entity_type='item'`. Mirrors `Item` with ISO strings instead of `Date`. |

`ListSnapshot` and `MemberSnapshot` are intentionally **not** added in
Phase 2. Phase-2 tools mutate items only; list/member-touching
activity_log rows land in Phase 3 (with `share_list` and member
management). Architect will add those types in the Phase 3 pass when
the executors that produce them ship.

## Cross-cutting invariants (apply to ALL six tools)

These rules are non-negotiable; an executor that breaks any of them
fails the Phase 2 verification gate. AI-agent's tool descriptions
should communicate the externally-visible parts (e.g. "ambiguous list
name resolves to Inbox") so the model emits inputs the executors can
satisfy.

### Inv-1: Transactional execution

Every executor wraps its DB work in a single Drizzle transaction. The
entity write **and** the activity_log write succeed together, or both
roll back. Half-applied state breaks the audit log and Phase-4 F2
restore.

```ts
// pseudo-shape every executor follows
return await db.transaction(async (tx) => {
  // 1. resolve list / item / membership
  // 2. mutate the entity (insert / update / soft-delete)
  // 3. insert activity_log row referencing actor + payload_before + payload_after
  // 4. return output object matching the tool's output schema
});
```

`list_lists` is the one read-only exception — no transaction, no
activity_log write.

### Inv-2: Membership check

Every item-touching tool (`create_item`, `update_item`,
`complete_item`, `delete_item`, `search_items`) first verifies the
calling user has `list_member` access to the target list with role
`'editor'` or `'owner'`. Viewer and non-members are rejected.

For `search_items`, the membership check is enforced at query time —
the query is constrained to lists the user has access to via
`list_members`. No "leak across users" path.

For mutation tools, on access failure the executor returns the
standard error envelope (Inv-4) with `code: 'forbidden'` and a
user-facing message AI-agent can render.

### Inv-3: List-id resolution (defensive)

`create_item` and `update_item` accept either a `list_id` (UUID) or a
human-readable `list_name`. Resolution order, applied by the executor
(NOT by the LLM):

1. If `list_id` is provided and the user is a member with edit role →
   use it. If the id resolves but the user lacks access → reject.
2. Else if `list_name` is provided:
   a. Exact case-insensitive match against the user's accessible
      lists → use it.
   b. Single fuzzy match (e.g. trigram / `ILIKE %name%`) with one
      candidate → use it.
   c. Multiple candidates → reject with `code: 'ambiguous_list'` and
      list the candidates in the error message; LLM will re-prompt
      the user.
3. Else (neither provided, or `list_name` matched zero lists) → fall
   back to the user's Inbox. Inbox is guaranteed to exist (created in
   `/start`).

This rule is the critical wedge feature: the bot should "just work"
even when the user is sloppy ("süt al"), but never silently pick the
wrong shared list.

### Inv-4: Error envelope

All executors return a discriminated result. AI-agent's `respond.ts`
formats failures as the tool's response message; LLM continues from
there.

```ts
type ExecResult<TOk> =
  | { ok: true; data: TOk }
  | { ok: false; error: { code: string; message: string } };
```

Common error codes (executors may add tool-specific ones):

| Code | When |
|---|---|
| `forbidden` | Membership check (Inv-2) failed |
| `not_found` | Item id / list id doesn't exist |
| `ambiguous_list` | List name matched multiple candidates |
| `invalid_input` | zod parse failed on the LLM-supplied input |
| `internal_error` | Unexpected DB / server failure |

### Inv-5: Activity log JSON contract (item mutations)

For every item-touching tool, the executor writes one `activity_log`
row inside the same transaction as the item mutation:

| Field | Value |
|---|---|
| `entity_type` | `'item'` |
| `entity_id` | the item's `id` |
| `list_id` | the item's `list_id` |
| `actor_id` | the calling user's id |
| `action` | one of the `ActivityAction` values (see below) |
| `payload_before` | `ItemSnapshot \| null` — full item row before the change, or `null` for `item_created` |
| `payload_after` | `ItemSnapshot \| null` — full item row after the change, or `null` for `item_deleted` (since we soft-delete by setting `archived_at`, this is the post-archive snapshot — see note) |

**Action mapping (Phase 2 subset):**

| Tool | Action(s) emitted |
|---|---|
| `create_item` | `item_created` |
| `update_item` | `item_edited` (text/position changes); `item_due_set` if dueAt was null and is now non-null; `item_due_cleared` if dueAt was non-null and is now null. **One activity row per call** — if multiple fields change, emit `item_edited` and capture all diffs in payload_before/after. The dedicated `item_due_*` actions are only used when `dueAt` is the SOLE field changed (helps Phase-3 reminder feed). |
| `complete_item` | `item_completed` if `is_done` becomes true; `item_uncompleted` if it becomes false |
| `delete_item` | `item_deleted` |

**`item_deleted` snapshot detail:** delete is soft (set `archived_at = now()`).
`payload_before` = the item as it existed pre-archive.
`payload_after` = `null`. This makes F2 restore trivial — read
`payload_before`, set `archived_at = null`, write a `list_restored`-equivalent
activity row (Phase 4 detail). Do NOT store the post-archive snapshot
in `payload_after`; it pollutes the diff view.

**`ItemSnapshot` serialization:** every `Date` field on `Item` becomes an
ISO 8601 string. Use `date.toISOString()`; never write `Date` objects
into a JSONB column directly (Drizzle will coerce, but the JSON
representation is implementation-defined and we want round-trip
stability for Phase-4 restore).

### Inv-6: Conversation history slicing

AI-agent owns `src/lib/ai/conversation.ts`. Slicing rule:

> Take the most-recent messages for `(user_id, chat_id)` ordered by
> `created_at desc`. Stop when EITHER:
> - 30 messages have been collected, OR
> - the cumulative content length exceeds **24,000 characters**
>   (≈6,000 tokens at the "4 chars per token" heuristic — under-counts
>   for non-Latin scripts but is the documented industry rule of
>   thumb).
> Whichever limit hits first. Then reverse to chronological order
> before sending to the LLM.

The 24,000-char budget includes `content` strings. `tool_calls` JSON
size is NOT counted — it's bounded (each tool call is small) and
counting it complicates the token estimate without changing the
outcome.

`MessageRole = 'tool'` rows carry the JSON-stringified executor output
in `content`. The slicer treats them like any other message; the LLM
needs the tool result in context to reason about follow-ups.

The slicing function signature (AI-agent freezes the exact shape):

```ts
function sliceConversation(
  messages: MessageWithToolCalls[],   // ordered desc by created_at
  opts?: { maxMessages?: number; maxChars?: number }
): ConversationMessage[];             // ordered asc, ready for LLM
```

Defaults: `maxMessages = 30`, `maxChars = 24_000`.

## Tool reference

For each tool: input schema (zod-shape pseudocode), output schema,
side effects, important rules.

Tool inputs/outputs are zod schemas in `src/lib/ai/tools.ts`
(AI-agent). Backend-agent's executor zod-parses the input defensively
on entry — never trust the LLM-supplied JSON without re-validation.

The error path is uniform across tools (Inv-4); each tool documents
only the success-path output.

### 1. `create_item`

> **LLM use case:** "süt al" / "okuma listesine Sapiens ekle" / "yarın
> 18:00 spor yap" → create one item.

#### Input

```ts
{
  text: string;                 // required, 1..2000 chars, trimmed
  list_id?: string;             // uuid; optional
  list_name?: string;           // optional; either id or name OR neither (→ Inbox)
  due_at?: string;              // ISO 8601; optional
  is_checkable?: boolean;       // default true (todo); false = note
}
```

If both `list_id` and `list_name` are supplied, `list_id` wins.
If neither resolves, fall back to Inbox (Inv-3).

#### Output

```ts
{
  item: ItemSnapshot;           // the freshly inserted row
  list: { id: string; name: string; emoji: string | null };
}
```

The output includes the resolved list info so the LLM can confirm
back to the user ("Süt'ü Inbox'a ekledim").

#### Side effects

- Insert into `items` (with `created_by = userId`, `position` = max+1
  in list, `is_done = false`).
- Insert into `activity_log` with `action='item_created'`,
  `payload_before = null`, `payload_after = ItemSnapshot`.

#### Rules

- Membership check (Inv-2) on the resolved list.
- List resolution per Inv-3.
- `due_at`, if present, must be in the future. Past dates → executor
  silently drops `due_at` (creates the item without a reminder); LLM
  sees a `warning` field in output. *Implementation note for
  Backend:* return `{ item, list, warnings: ['due_at_in_past'] }`.
  AI-agent: include `warnings: string[]` (optional) on the output
  schema.
- `is_checkable=false` (note) implicitly forces `is_done=false` and
  ignores any `due_at` (notes don't fire reminders). Validate at
  parse time.

### 2. `search_items`

> **LLM use case:** "okuma listemde Sapiens var mı" / "süt eklemiş
> miydim" / "alışveriş listesinde ne var" → return matching items
> with list context. Cross-list when no list specified.

#### Input

```ts
{
  query: string;                // required, 1..500 chars
  list_id?: string;             // uuid; if set, scope to that list
  list_name?: string;           // optional; same resolution as create
  include_done?: boolean;       // default false — by default skip completed items
  include_archived?: boolean;   // default false — soft-deleted items hidden
  limit?: number;               // default 20, max 50
}
```

`list_id` and `list_name` both being absent = search across ALL lists
the user is a member of (cross-list mode). This is the common case
("Sapiens'i ekledim mi" doesn't say which list).

#### Output

```ts
{
  results: Array<{
    item: ItemSnapshot;
    list: { id: string; name: string; emoji: string | null };
    /** Trigram / ILIKE match score 0..1; used by LLM to rank in reply.
     *  Phase 2 implementation can return 1 for all matches; Phase 4 may
     *  upgrade to pg_trgm similarity. */
    score: number;
  }>;
  total_matched: number;        // may exceed results.length when capped at limit
  searched_lists: Array<{ id: string; name: string }>;  // which lists were scanned
}
```

#### Side effects

None. Read-only.

#### Rules

- Membership check (Inv-2) per scanned list — query joins
  `list_members` on `user_id = caller`.
- Phase 2 search = `ILIKE %query%` on `items.text` per the architecture
  doc ("Postgres ILIKE on `items.text` … sufficient for Phase-1 list-scoped
  search"). LLM expectation: this is fuzzy-ish but not semantic.
  Phase 2+ pgvector is out of scope.
- `searched_lists` is included so the LLM can render contextual
  responses ("Sapiens'i Okuma listende buldum") without a second
  round-trip.

### 3. `update_item`

> **LLM use case:** "süt yerine yoğurt yaz" / "Sapiens'i hafta sonuna
> çek" / "spor saatini 19'a al".

#### Input

```ts
{
  item_id: string;              // uuid, required
  text?: string;                // 1..2000 chars
  due_at?: string | null;       // ISO 8601, or explicit null to clear
  position?: number;            // for drag-style reorder via natural language
}
```

At least one of `text`, `due_at`, `position` must be supplied; else
`invalid_input`.

`due_at: null` is meaningfully different from omitting `due_at` — the
former clears the reminder; omitting leaves it untouched. AI-agent's
schema must distinguish.

#### Output

```ts
{
  item: ItemSnapshot;           // post-update
  changes: Array<'text' | 'due_at' | 'position'>;  // which fields actually changed
}
```

`changes` lets the LLM produce specific confirmation copy ("Metni
değiştirdim" vs "Hatırlatmayı kaldırdım").

#### Side effects

- Update `items` row.
- Insert one `activity_log` row. Action = `item_edited` if text or
  position changed; `item_due_set` / `item_due_cleared` only when
  `due_at` is the *sole* field that changed (Inv-5 mapping).

#### Rules

- Membership check on the item's list (Inv-2).
- If new `due_at` is in the past, executor silently drops the change
  for that field and surfaces a warning, identical to `create_item`.
- `text` is trimmed; empty post-trim → `invalid_input`.

### 4. `complete_item`

> **LLM use case:** "süt'ü işaretle" / "spor'u tamamladım" / "Sapiens
> okumayı bitirdim mi diye işaretsiz yap".

#### Input

```ts
{
  item_id: string;              // uuid, required
  is_done: boolean;             // EXPLICIT — true to complete, false to uncomplete
}
```

Explicit `is_done` (rather than a "toggle") avoids ambiguity when the
LLM mis-reads state. AI-agent's tool description must instruct the
model: "before calling, decide explicitly whether the user wants the
item done or not done".

#### Output

```ts
{
  item: ItemSnapshot;
  was_done: boolean;            // previous state — useful for "already done" replies
}
```

If `was_done === input.is_done`, the executor still runs (no error),
writes no activity_log row, and returns the unchanged item. AI-agent
should render this as "Already marked done" rather than a redundant
confirmation.

#### Side effects

- Update `items.is_done` and `items.completed_at`
  (`completed_at = now()` when transitioning false→true; cleared when
  true→false).
- Activity_log: `item_completed` (false→true) or `item_uncompleted`
  (true→false). **No activity row when state didn't change.**

#### Rules

- Membership check (Inv-2).
- Notes (`is_checkable=false`) cannot be completed → `invalid_input`.

---

### 5. `delete_item`

> **LLM use case:** "süt'ü sil" / "Sapiens'i listeden çıkar".

#### Input

```ts
{
  item_id: string;              // uuid, required
}
```

#### Output

```ts
{
  item: ItemSnapshot;           // the item as it was just before archiving
}
```

Returning the snapshot lets the LLM offer "undo" copy ("Sapiens'i
sildim. Geri almak ister misin?") without an extra fetch.

#### Side effects

- Soft-delete: set `items.archived_at = now()`.
- Activity_log: `item_deleted`, `payload_before = ItemSnapshot`,
  `payload_after = null` (per Inv-5).

#### Rules

- Membership check (Inv-2).
- Already-archived items → `not_found` (we don't expose archived items
  to the model; the LLM can't accidentally double-delete).
- Hard delete is NEVER performed by tools; only Phase-4 retention
  jobs hard-delete after the 30-day restore window.

### 6. `list_lists`

> **LLM use case:** "hangi listelerim var" / "okuma listemde kaç tane
> kitap var". Read-only; the LLM uses this when it needs to enumerate
> before deciding which list a request targets.

#### Input

```ts
{
  include_archived?: boolean;   // default false
}
```

(No filter args needed in Phase 2; sharing/role filters land in Phase 3.)

#### Output

```ts
{
  lists: Array<{
    id: string;
    name: string;
    emoji: string | null;
    is_inbox: boolean;
    role: ListRole;             // 'owner' | 'editor' | 'viewer' from list_members
    item_count: number;         // total items, archived excluded, completed excluded by default
    open_count: number;         // items where is_done=false AND archived_at is null
  }>;
}
```

`item_count` and `open_count` are computed via a single LEFT JOIN
aggregation — do not N+1 per list.

#### Side effects

None. Read-only; no activity_log.

#### Rules

- Returns ALL lists where the caller is a member (any role).
- Inbox is always present (created on `/start`). It should appear
  first in the array — order: inbox first, then by `created_at asc`.
- `archived_at IS NOT NULL` lists hidden unless `include_archived=true`.

## Bot ↔ AI ↔ Executor flow (reference)

```
Telegram Update
    │
    ▼
[Backend] webhook handler           src/app/api/telegram/webhook/route.ts
    │  - verify secret, parse Update, ack 200
    │  - look up user
    ▼
[Backend] handle-message.ts         src/lib/server/bot/handle-message.ts
    │  - load last 30 messages from messages table → MessageWithToolCalls[]
    │  - decrypt user.openrouter_api_key_encrypted
    ▼
[AI] respond.ts                     src/lib/ai/respond.ts
    │  - sliceConversation() per Inv-6
    │  - assemble system prompt (locale-aware) + sliced history + new user turn
    │  - call Anthropic SDK (baseURL = OpenRouter) with tool schemas from tools.ts
    │  - if model returns tool_calls:
    │       loop: call each Backend executor → collect ToolResult
    │             feed results back to LLM → potentially more tool calls
    │       (cap: 5 tool round-trips per turn — guard against runaway)
    │  - return final assistant message
    ▼
[Backend] handle-message.ts (cont)
    │  - persist user + assistant + tool messages to DB (in order)
    │  - sendMessage() to Telegram
```

The **5-round-trip cap** is a safety belt; well-behaved tool sequences
finish in 1–2. Hitting the cap returns the partial result + a system
note to the user ("Bir şeyler ters gitti, tekrar dener misin?").

## Conversation persistence rules

Backend-agent owns `src/lib/db/queries/messages.ts`:

- After every bot turn, persist messages to `messages` table in
  chronological order:
  1. one row, role='user', content = raw user text
  2. zero-or-more pairs (assistant-with-tool_calls + tool-result(s))
     — one row per LLM tool-calling step, in the order the LLM
     emitted them
  3. one row, role='assistant', the final user-facing reply
- `tool_calls` column receives the raw `ToolCall[]` JSON when the
  assistant message contains tool calls; `null` otherwise.
- `tool_call_id` column (on role='tool' rows) references the
  originating ToolCall.id so future reads can pair them.
- All four message types share the same `chat_id` and `user_id`.
- `messages` is append-only — no UPDATE / DELETE in Phase 2 (the
  `/reset` command in Phase 3 will DELETE-WHERE).

`MessageWithToolCalls` (the parsed shape) is what the slicer and
respond.ts consume; the query helper does the JSONB→`ToolCall[]`
conversion at the read boundary.

## BYOK key handling (notes for Backend)

Out of Architect scope, but the contract intersects:

- `users.openrouter_api_key_encrypted` is the AES-256-GCM ciphertext
  (Backend encrypts on `PATCH /api/settings`).
- AI-agent's `respond.ts` accepts the *decrypted* key as a function
  argument; AI-agent never reads the env or DB directly.
- If decryption fails or the field is null, AI-agent returns a
  sentinel "no key configured" assistant message; Backend formats it
  for the user ("Add your OpenRouter API key in Mini App settings to
  use AI features.").
