### P2-3: Cron heartbeat fires on partial-failure ticks (per-row DM failure â‰  unsuccessful run)

**Where**: `src/lib/cron/dispatch-reminders.ts:234-244`. `main()` calls
`maybePingHeartbeat()` whenever `dispatchReminders()` returns without
throwing. Per-row DM failures don't throw (correctly â€” Inv-11 says
to retry next tick), so a tick that picked 5 items and failed 5 DMs
still pings the heartbeat as "successful run".

**Why minor**: the heartbeat is monitoring, not load-bearing. A pure
"is the cron container alive and reaching Postgres" signal is still
useful (which is what the current behavior surfaces). But operators
reading the Better Stack heartbeat's intent literally would expect a
miss when delivery is broken.

**Recommendation**: in Phase 4 (when `cron-heartbeat target` open
question is fully resolved), either (a) accept the current semantics
and document them in the heartbeat env var's comment, or (b) gate
the ping on `result.failed === 0 || result.sent > 0` so a 100%-failed
tick suppresses the heartbeat. (a) is simpler and matches the
"liveness, not health" framing in `architecture-pass-phase-3.md`.

**Severity**: P2 â€” semantic ambiguity; no security impact.

**Route**: Backend + Architect (Phase 4 pass).

### P2-8: Cron dispatcher does not implement Inv-11's "â‰Ą3 consecutive failures â†’ log `reminder_send_persistent_failure`" warning

**Where**: `src/lib/cron/dispatch-reminders.ts:204-218`. Per-row
failure is logged with `console.error` and counted in `failed`, but
no per-item failure-streak counter is maintained. The Architect's
contract says: "After â‰Ą3 consecutive failures for the same item, the
dispatcher logs a warning (`reminder_send_persistent_failure`) but
**does not** mark `reminder_sent = true` â€” operator must intervene.

of the cron container."

**Why minor (and arguably moot)**: the cron container in Dokploy's
"once per tick" model exits after each `npm run cron` invocation, so
in-memory state across ticks is impossible by construction. The
intent of the warning (operator visibility for stuck items) is sound
but the implementation surface needs a different mechanism â€” either
a `reminder_failure_count` column (schema is frozen â€” Phase 4
amendment) or persisting the streak in `activity_log`.

**Recommendation**: defer to Phase 4 architecture pass; pick one of:
(a) acknowledge the contract gap and remove the requirement (cron
container is per-tick, not long-running); (b) add a
`reminder_failure_count` column in Phase 4; (c) detect persistent
failure post-hoc by querying `WHERE due_at < NOW() - interval '5
minutes' AND reminder_sent = false`. Option (a) + (c) together is the
lowest-cost path.

**Severity**: P2 â€” operator-observability gap; reminders still get
retried correctly per Inv-11.

**Route**: Architect (Phase 4 contract amendment) + Backend (after
ratification).
