## 4. Findings — important (P1)

### P1-1: `GET /api/settings` is missing `hasApiKey`; Frontend reads `initial.hasApiKey` and silently always renders "no key stored"

**Where**: `src/app/api/settings/route.ts` GET handler (lines 58-64) returns
`{ locale, timezone, llmModel, notificationsEnabled, byokKeyPreview }`.

**Frontend contract**: `src/components/settings/settings-form.tsx` line 57
declares `SettingsInitial.hasApiKey: boolean` and uses it at line 167:
`hasStoredKey={initial.hasApiKey}`. The settings page server-fetches the
API and casts the response to `SettingsInitial` (`src/app/(app)/settings/page.tsx`
line 65), so `hasApiKey` is `undefined` → coerces to falsy → `<ApiKeyField>`
always renders the "no stored key" branch even when the user has saved one.
Visually: returning users see an empty input with placeholder "OpenRouter
API key" instead of the "Stored: sk-••••XXXX [Replace] [Remove]" affordance.

**Recommendation**: Backend adds `hasApiKey: !!user.openrouterApiKeyEncrypted`
to the GET response (and PATCH response, same shape). Validator's
`GetSettingsResponse` type updates accordingly. No DB migration; no Frontend
change needed.

**Severity**: P1 — Phase 2 acceptance test #4 ("first LLM call uses new
key") still passes, but the visible UI affordance for re-using a saved key
is broken. Self-healing because `<ApiKeyField>` driven by
`storedKeyPreview={initial.byokKeyPreview}` would still show the preview
when the editing branch is taken — except the form forces editing mode
when `!hasStoredKey` (api-key-field.tsx line 45). Net effect: confusing UX
for returning settings visits.

**Route**: Backend.

### P1-2: `PATCH /api/items/[id]` response shape doesn't match the `apiPatch<Item>` Frontend typing

**Where**: `src/app/api/items/[id]/route.ts` returns the raw executor
envelope: `{ ok: true, data: { item: ItemSnapshot, was_done?, changes?, warnings? } }`
(via `NextResponse.json(lastResult)` at line 131).

**Frontend contract**: `src/components/lists/item-list.tsx` lines 67, 98, 164
all type the mutation as `apiPatch<Item>` and the `useMutation` generic
`TData` is also `Item`. After the api-client envelope unwrap (api-client.ts
lines 36-39), the resolved value is `data` — i.e., `{ item: ItemSnapshot,
was_done?, ... }`, NOT `Item`.

**Why this hasn't broken anything yet**: the optimistic UI (`onMutate`
patches the cache, `onSettled` invalidates and refetches) doesn't read
the mutation's resolved value — `onSuccess`/`onSettled` ignore `data`.
So the type lie is invisible at runtime today. It will bite the moment
any future mutation handler does `onSuccess: (item) => ...`.

**Recommendation**: pick one:
1. Change Frontend to `apiPatch<{ item: Item }>` and adjust call sites
   (small; matches reality).
2. Change Backend PATCH route to unwrap to just the item: `NextResponse.json({ ok: true, data: lastResult.data.item })`.
   Simpler client side but loses `changes`/`warnings`/`was_done`
   metadata that the bot path receives.

Backend's tool dispatcher uses the full envelope; the Mini App route
should NOT diverge from the bot in shape — option 1 keeps surfaces
parallel. Recommendation: option 1 (Frontend-side typing fix).

**Severity**: P1 — typing bug rather than runtime bug today, but the
Frontend ↔ Backend contract drifted from the agents.md "Backend exports
the inferred type; Frontend imports type" rule. Surface it now before
Phase 3 builds on it.

**Route**: Frontend (validator/type fix only) — though Backend should
also export a `PatchItemResponse` type from `src/lib/validators/items.ts`
to make the contract explicit.

### P1-3: `list_lists` `item_count` semantic divergence from contract

**Where**: `src/lib/server/tools/list-lists.ts` lines 86-87:

```ts
item_count: r.totalCount,    // count of all non-archived items (incl. completed)
open_count: r.openCount,     // count of non-archived AND is_done=false
```

**Architect contract** (`docs/architecture-pass-phase-2.md` § 6, list_lists output):

```
item_count: number;   // total items, archived excluded, completed excluded by default
open_count: number;   // items where is_done=false AND archived_at is null
```

Per the spec, `item_count` is "completed excluded by default" — i.e., it
should equal `open_count`. The implementation surfaces total
non-archived (including done) under `item_count`. Backend's own report
flagged this; Backend chose to expose both because the spec wording is
ambiguous and "we expose both" felt useful.

**Recommendation**: align with the Architect's contract — both `item_count`
and `open_count` should be the open count. If Backend wants a "total
including done" surface, that's a Phase 3+ contract amendment via
Architect, not in-flight code drift.

**Severity**: P1 — schema-level contract divergence between AI-agent's
LLM-facing tool schema (it advertised the Architect's spec to the model)
and the executor's actual return values. The LLM may render incorrect
counts ("Okuma listende 5 öğe var" when 3 are done and 2 open). Visible
to end users.

**Route**: Backend (one-line change: `item_count: r.openCount`).

## 5. Findings — minor (P2)

### P2-1: `search_items` with `list_id`/`list_name` rejects viewer-role members

**Where**: `src/lib/server/tools/search-items.ts` calls `resolveList`
which checks `WRITE_ROLES` (`_shared.ts:131`) and returns
`forbidden` for viewer members. The cross-list branch (no
list_id/list_name) correctly allows ANY membership role.

**Why minor in Phase 2**: there are no viewers in Phase 2 — sharing +
roles ship in Phase 3. Surfacing now so Phase 3 doesn't trip over it.

**Recommendation**: when `_shared.resolveList` is called from
`search-items` (which is read-only), allow `viewer` role too. Easiest:
add `opts.allowViewer?: boolean` to `resolveList`, set true from
`search-items.ts`. Alternatively, search-items duplicates the list
resolution inline without going through `resolveList`.

**Route**: Backend (Phase 3 — defer).

### P2-2: PATCH `/api/items/[id]` when both `isDone` and `text/dueAt/position` are present runs two transactions

**Where**: `src/app/api/items/[id]/route.ts` lines 82-116 dispatch
`executeUpdateItem` then `executeCompleteItem` sequentially when the
client body contains both kinds of edits. The route's own comment at
line 73 acknowledges this trade-off.

**Why minor**: the Mini App today never sends a combined body
(`toggleMutation` sends only `{ isDone }`; `editMutation` sends
`text`/`dueAt` only; `reorderMutation` sends `{ position }` only). The
risk is theoretical until a future feature combines them. The trade-off
the route makes (two transactions vs. a more complex single executor)
is sane.

**Recommendation**: add a defensive comment that the bot path doesn't
combine these either; document the contract that future combined edits
must use a different surface.

**Route**: Backend (no immediate action).

### P2-3: `console.error("[bot/handle-message] respond() threw", error)` could log Anthropic SDK auth-error envelopes containing the API key in headers

**Where**: `src/lib/server/bot/handle-message.ts` line 151.

**Risk**: when Anthropic SDK throws on a bad API key, the thrown error
object can include `request` metadata. A naive `console.error(err)` may
or may not stringify the request headers. Anthropic SDK v0.x typically
strips Authorization from error logs, but this is environment-dependent.

**Recommendation**: replace with `console.error("[bot/handle-message] respond() threw", error instanceof Error ? error.message : String(error))`
to surface the message only. Better still: route to Sentry with a hand-
selected context (Phase 4 OSS pass).

**Route**: Backend (Phase 4 — Sentry wiring).

### P2-4: `list_lists` SQL builds `itemCount` and `openCount` with the IDENTICAL `filter (where archived_at is null AND is_done = false)` predicate

**Where**: `src/lib/server/tools/list-lists.ts` lines 44-45.

Both `itemCount` and `openCount` columns are computed with the same
filter clause. This is harmless (the duplicate computation runs in the
same SQL aggregate pass), but the inline comment claims `itemCount` is
"all items" — it isn't. The actual code sets `item_count: r.totalCount`
(line 86) which IS computed correctly elsewhere (line 50). The SQL
duplicate is dead code waiting to confuse a future maintainer.

**Recommendation**: drop the duplicate `itemCount` SQL aggregate and
just use `totalCount`. Or, after fixing P1-3, drop `totalCount` and
keep only the open one. Either way, the dead aggregate should go.

**Route**: Backend (housekeeping).

### P2-5: Missing test for `src/lib/server/encryption.ts` round-trip

**Where**: encryption.ts comment says "Phase 4 will add a Vitest round-
trip suite". Phase 2's pragmatic verification gate doesn't require unit
tests. Calling out for Phase 4 reviewer to confirm.

**Route**: Reviewer (Phase 4).

### P2-6: `webhook/route.ts` 401 on bad secret could leak the existence of the endpoint to scanners

**Where**: `src/app/api/telegram/webhook/route.ts:14-19` returns 401 with
a JSON envelope `{ ok: false, error: { code: 'unauthorized', message: 'Invalid webhook secret' } }`.

**Why minor**: Telegram never hits the endpoint without the header (it
sends the configured `secret_token`); legitimate Telegram requests
never see the 401. Returning a generic 404 (or 200 with no body) on
mismatch would reduce fingerprinting surface. Convention varies; the
current implementation is defensible.

**Recommendation**: out of scope for Phase 2; consider in Phase 4 OSS
hardening pass.

**Route**: Reviewer (Phase 4).
