# Architecture Pass — Phase 3 (Sharing + Reminders + Assignments)

> Generated by Architect-agent on 2026-05-02.
> This is the canonical contract for AI-agent (`src/lib/ai/**`) and
> Backend-agent (`src/lib/server/**`, `src/lib/cron/**`,
> `src/lib/validators/**`, `src/app/api/**`) for Phase 3, plus the
> read-side contract Frontend-agent consumes for the share sheet,
> invite-accept screen, member management, assignee badge, and B1
> activity feed. Any deviation = Phase 3 verification failure;
> deviations require a new Architect invocation, not in-flight
> hot-fixes.

## Scope

Phase 3 ships three new LLM tools — `share_list`, `schedule_reminder`,
`assign_item` — plus the supporting persistence and cron flows that
bring listgram from "single-user list assistant" to "shared,
time-aware, assignable list assistant":

1. **Sharing** — `share_list` tool (LLM path) and `/share` slash
   command (deterministic path) both create a `list_invites` row +
   send a Telegram deeplink DM to the invitee. The invitee taps the
   deeplink → `/app/invites/[token]` accept screen → `POST
   /api/invites/[token]/accept` writes a `list_members` row in the
   same transaction that flips the invite to accepted.

2. **Reminders** — `schedule_reminder` tool sets `items.due_at` (it
   is a thin wrapper over `update_item` with semantic clarity).
   `src/lib/cron/dispatch-reminders.ts` runs every 60 s in a Dokploy
   cron container; it picks rows via the existing
   `items_due_at_idx` partial index, DMs the owner (or assignee, if
   set), then flips `reminder_sent = true`.

3. **Assignments** — `assign_item` tool resolves an `@username` or
   bare first name against the target list's members (lowered
   `telegram_username` exact match; case-insensitive prefix match on
   `telegram_first_name` as fallback), then writes
   `items.assignee_id` plus an `item_assigned` activity row.

4. **B1 activity feed** — `GET /api/lists/[id]/activity` exposes
   the existing `activity_log` rows (already written by Phase 2
   tools) as a paginated list-scoped feed, joined with actor user
   info for client-side rendering.

What is NOT in this phase (deferred to Phase 4): forwarded-message
handling A3, inline mode D1, shareable list snapshot D2, audit/restore
F2, full data export F1, marketing landing rebuild, E2 a11y audit,
E1/E3 i18n bot multilingual hardening, Docker Compose. See § 9.

The DB schema is **frozen** since Phase 1: every column Phase 3 needs
already exists (`list_invites.token / expires_at / accepted_at /
accepted_by_user_id / role`, `items.assignee_id / due_at /
reminder_sent`, `list_members.role`, all relevant
`activity_log.action` enum values, `items_due_at_idx` partial index).
Architect did not — and will not — re-touch `src/lib/db/schema.ts`.

### 7. `share_list`

> **LLM use case:** "okuma listesini @ali ile paylaş" / "alışveriş
> listesine kardeşimi ekle, kullanıcı adı ahmetk" → create a `list_invites`
> row and send a Telegram deeplink DM.

#### Description (LLM-facing)

> "Invite another Telegram user to one of the lists you own as an
> editor (default) or viewer. Pass `username` (with or without the
> leading @ — case is normalized). Pass `list_id` when known; else
> `list_name` (resolved with the same rules as `create_item`). The
> bot DMs the invitee a deeplink that, when tapped, opens the Mini
> App accept screen. The invitee must have started the bot at least
> once for the DM to land. If the user is already a member, the
> tool returns `already_member` and does not create a duplicate
> invite."

#### Input

```ts
{
  username: string;             // 1..32 chars; lowered + leading @ stripped
  list_id?: string;             // uuid; optional
  list_name?: string;           // optional; same Inv-3 resolution
  role?: 'editor' | 'viewer';   // default 'editor'
}
```

At least one of `list_id` or `list_name` must be supplied. **Inbox
fallback is NOT applied to `share_list`** — sharing your Inbox is
nonsensical (it's per-user). If neither resolves, the executor returns
`invalid_input`.

#### Output

```ts
{
  invite: {
    token: string;              // 32-byte URL-safe (base64url)
    expiresAt: string;          // ISO 8601, default now + 7 days
    deeplink: string;           // e.g. "https://t.me/<bot>?startapp=invite_<token>"
    role: 'editor' | 'viewer';
  };
  list: { id: string; name: string; emoji: string | null };
  invitedUsername: string;      // lowered, no @ — what was stored
  /** True when the invitee is already a member; no invite was created. */
  alreadyMember?: boolean;
}
```

When `alreadyMember === true`, `invite` is still populated **but**
references an existing/synthetic record only for shape consistency;
AI-agent should phrase the reply as "Ali zaten bu listenin üyesi" and
skip the deeplink. Backend implementation may return the most-recent
existing (non-expired, non-accepted) invite if present, or no invite
at all when `alreadyMember`.

#### Side effects

- INSERT `list_invites` row with `token = crypto.randomBytes(32).toString('base64url')`,
  `expires_at = now() + interval '7 days'`, `invited_username = lowered`,
  `role` (defaults to `'editor'`).
- Send Telegram DM to the invitee via grammY: list emoji + name + "X
  invited you" + deeplink button. (DM dispatch is best-effort: if the
  invitee has not started the bot, Telegram returns 403 — the executor
  surfaces a `warning` ('invitee_dm_failed') but the invite row remains
  valid, so they can still receive the deeplink via copy-paste.)
- No item mutation. Activity log: writes one row with
  `entity_type='member'`, `entity_id = <pre-allocated future
  list_member id, computed via uuid_generate>` is **not** the chosen
  approach — instead, the `member_added` activity row is written at
  **accept time** (Backend's `accept-invite.ts`), where the `list_members`
  row is actually created. `share_list` itself emits **no activity row**
  — it is invite-creation, not membership change. (See Inv-13.)

#### Rules

- Caller must be `owner` of the target list (only the owner can
  invite). Editors and viewers cannot share. → `forbidden`.
- `username` lowered + leading `@` stripped before storage and
  comparison. The `users.telegram_username` column has a lower-cased
  index (`users_telegram_username_idx`); searches use `lower(...)`.
- `username` self-invite (caller's own username) → `invalid_input`.
- If a non-expired, non-accepted invite already exists for
  `(list_id, lowered_username)`, **reuse** it (return its existing
  token + expiresAt) rather than creating a duplicate. This is the
  idempotency path: re-tapping `/share` after sending should not
  produce a forest of pending invites.
- The Inbox cannot be shared (`is_inbox = true`) → `invalid_input`
  with code `cannot_share_inbox`.
- Writes are transactional: invite row + DM send attempt happen in
  sequence; DM failure does NOT roll back the invite (Inv-13).

### 8. `schedule_reminder`

> **LLM use case:** "yarın saat 18'de spor yapmamı hatırlat" /
> "Sapiens'i pazartesi 09:00'da hatırlatsın" → set or clear a due
> date on an existing item.

#### Description (LLM-facing)

> "Set, change, or clear an item's reminder time. Pass `item_id` and
> either `due_at` (ISO 8601 in the future) to schedule, or
> `due_at: null` (explicit) to clear. The reminder fires as a
> Telegram DM at the given time (UTC-aligned within ±60 s). If the
> item has an assignee, the DM goes to the assignee; otherwise to
> the item's creator. Past times are silently dropped with a
> warning. If you only want to create a fresh item with a reminder,
> call `create_item` with `due_at` instead — `schedule_reminder` is
> for already-existing items."

#### Input

```ts
{
  item_id: string;              // uuid, required
  due_at: string | null;        // ISO 8601, OR explicit null to clear
}
```

`due_at: null` is meaningfully different from omitting it — the
former clears the reminder. The schema requires `due_at` to be
present (string or null); omission → `invalid_input`. This
distinguishes `schedule_reminder` semantically from `update_item`
even though their backing column is the same.

#### Output

```ts
{
  item: ItemSnapshot;           // post-update; reminder_sent reset to false
  cleared: boolean;             // true when due_at was set non-null and is now null
  /** Soft warnings (e.g. due_at_in_past). */
  warnings?: string[];
}
```

#### Side effects

- UPDATE `items` row: `due_at = <new>`, `reminder_sent = false`
  (re-arms the cron pickup even when re-scheduling an already-fired
  reminder).
- Activity log: action `item_due_set` when transitioning to a non-null
  due_at; `item_due_cleared` when transitioning to null. **One row per
  call.** No row written if `due_at` is unchanged (e.g. LLM re-scheduled
  to the same value) — same no-op rule as `complete_item`.

#### Rules

- Membership check (Inv-2) on the item's list — caller must be
  `owner` or `editor`.
- Notes (`is_checkable=false`) cannot have due_at → `invalid_input`
  with code `cannot_schedule_note`.
- Past `due_at` values are silently dropped: the executor writes
  nothing, returns `{ item: <unchanged-snapshot>, cleared: false,
  warnings: ['due_at_in_past'] }`. AI-agent should re-prompt the
  user for a future time.
- Re-arming an already-fired reminder (item where `reminder_sent =
  true`) is supported — the executor sets `reminder_sent = false`
  and the cron picks it up on the next tick.

### 9. `assign_item`

> **LLM use case:** "@ali süt'ü sen al" / "Sapiens'i kardeşim okusun"
> / "alışveriş listesindeki yumurta'yı bana ata" → set or clear an
> item's assignee.

#### Description (LLM-facing)

> "Assign an item to one of the list's members, or unassign it.
> Pass `item_id` and either `assignee_username` (with or without @)
> to assign, or `assignee_username: null` (explicit) to unassign.
> The username resolves against the list's members only — you
> cannot assign to a non-member; if you try, the tool returns
> `not_a_member` and you should call `share_list` first. If
> `assignee_username` matches multiple members ambiguously (e.g.
> two users named 'Ali' and only first names are given), the tool
> returns `assignee_ambiguous` with candidates — ask the user to
> clarify."

#### Input

```ts
{
  item_id: string;              // uuid, required
  assignee_username: string | null;
  // username (with/without @) to assign; null = unassign.
}
```

`assignee_username` accepts either a leading-@ Telegram handle
("@ali" → matched against `lower(telegram_username)`) OR a
case-insensitive bare token ("ali", "Ali", "ALI") matched against
`lower(telegram_first_name)` as a fallback. The executor — not the
LLM — does this resolution.

#### Output

```ts
{
  item: ItemSnapshot;           // post-update
  /** The user the item is now assigned to, or null if unassigned. */
  assignee: {
    id: string;
    telegramUsername: string | null;
    telegramFirstName: string;
  } | null;
  previousAssigneeId: string | null;
}
```

#### Side effects

- UPDATE `items.assignee_id`.
- Activity log: action `item_assigned` (assigning, including
  re-assigning A→B); `item_unassigned` (clearing); **no row when
  assignee_id unchanged**.

#### Rules

- Membership check (Inv-2) on the item's list — caller must be
  `owner` or `editor`.
- The assignee being set MUST be a member of the same list (Inv-12).
  Resolution is membership-scoped: the executor JOINs `list_members`
  and only matches users with a row for that list. Non-member
  matches → `not_a_member` (with a hint: "Önce `/share` ile davet
  et").
- Resolution rules:
  1. Strip leading `@`; lower-case.
  2. Exact match against `lower(telegram_username)` of list members
     → exactly one hit → use it.
  3. Else, case-insensitive prefix match against
     `lower(telegram_first_name)` of list members → exactly one hit
     → use it.
  4. Multiple hits at either step → `assignee_ambiguous` with
     candidates ({ id, telegramUsername, telegramFirstName }) so the
     LLM can ask "Ali'lerden hangisi?" — the LLM then re-calls with
     the disambiguating handle.
  5. Zero hits → `not_a_member`.
- Self-assign is allowed (assigning to caller).
- Notes (`is_checkable=false`) **can** be assigned (note ownership is
  meaningful, e.g. "this idea is Ali's").

## Re-confirmation of Phase-2 invariants

The Phase 2 invariants Inv-1..Inv-9 apply unchanged in Phase 3.
Restated for clarity (full text in
`docs/architecture-pass-phase-2.md`):

- **Inv-1** Transactional execution — every executor wraps DB writes
  in a single Drizzle transaction; entity write + activity_log write
  succeed together or roll back together. `list_lists` and the new
  read-only invite/activity GETs are the only exceptions.
- **Inv-2** Membership check — every list-touching tool verifies the
  caller's `list_members` role before acting. `WRITE_ROLES =
  ['owner', 'editor']`. `share_list` tightens this to **owner-only**;
  Phase-3 read endpoints (activity feed, member list) accept any role.
- **Inv-3** List-id resolution — `list_id` > exact name > fuzzy
  name > Inbox fallback (when applicable; `share_list` does NOT
  apply Inbox fallback). Multi-fuzzy → `ambiguous_list`.
- **Inv-4** Error envelope — `{ ok: true, data } | { ok: false,
  error: { code, message } }` returned from every executor. Common
  codes: `forbidden`, `not_found`, `ambiguous_list`,
  `invalid_input`, `internal_error`. Phase-3 codes added below
  (Inv-14).
- **Inv-5** Activity log JSON contract — `entity_type`, `entity_id`,
  `list_id`, `actor_id`, `action`, `payload_before`, `payload_after`
  written inside the same transaction. Phase 3 extends the snapshot
  union to include `ListSnapshot` and `MemberSnapshot`.
- **Inv-6** Conversation history slicing — last 30 messages OR
  24,000 chars cumulative; whichever first. Unchanged.
- **Inv-7** Append-only `messages` — Phase 3 introduces `/reset` as
  the **only** DELETE-WHERE on the table; it deletes by
  `(user_id, chat_id)`. Slicer behavior unchanged on a fresh chat.
- **Inv-8** BYOK key flow — encrypted at rest, decrypted ephemerally
  per turn, never logged, passed as a function argument to AI's
  `respond.ts`. Unchanged.
- **Inv-9** Webhook-200-on-any-internal-error — webhook handler
  always acks 200 within 60 s; LLM/tool errors caught upstream of
  the response. Unchanged.

### New invariants

#### Inv-10: Invite token entropy

`list_invites.token` is generated as
`crypto.randomBytes(32).toString('base64url')` — 32 bytes (256 bits)
of CSPRNG entropy, encoded URL-safe. Tokens are unique (DB constraint
already exists) and **case-sensitive** at compare time (no lower-casing).
Token length under base64url encoding is 43 chars.

The token doubles as the only authn surface for the invite-accept
GET (the invitee may not yet have a session). Therefore:

- The accept POST DOES require an authenticated session (the user
  must have completed Telegram initData verification first; the
  Mini App middleware enforces this).
- Tokens are not enumerable nor brute-forceable in any meaningful
  time at 256-bit entropy.
- Tokens MUST NOT be logged. (Backend `console.log` of invite
  payloads must mask the token, similar to BYOK key handling.)

#### Inv-11: Reminder dispatcher idempotency

The cron dispatcher achieves exactly-once-ish DM delivery via the
`items.reminder_sent` boolean:

- Pickup query filters `WHERE due_at <= NOW() AND reminder_sent =
  false AND archived_at IS NULL` (this is exactly the partial-index
  predicate from Phase 1's `items_due_at_idx`).
- For each picked row, the dispatcher attempts the DM, then runs
  `UPDATE items SET reminder_sent = true WHERE id = $1 AND
  reminder_sent = false` **only on send success**. The
  `reminder_sent = false` predicate in the WHERE is a defensive
  guard against double-fire when two cron containers race (Dokploy
  default = single instance, but the guard is cheap).
- DM send failure → leave `reminder_sent = false`; the next minute's
  tick will retry. After ≥3 consecutive failures for the same item,
  the dispatcher logs a warning (`reminder_send_persistent_failure`)
  but **does not** mark `reminder_sent = true` — operator must
  intervene. Failure backoff is per-item count tracked in-memory
  for the lifetime of the cron container (no DB column needed for
  Phase 3).
- Cron run completes → optional GET on `LISTGRAM_HEARTBEAT_URL` env
  var if set, else no-op. (Heartbeat is opt-in; resolves the open
  question in `handoff/specs/CLAUDE.md`. Default-on/public-endpoint
  rejected because it implies a managed dependency.)

Schedule-jitter / clock-skew handling: the picker uses `NOW()`
server-side (Postgres clock), not the cron container clock — this
removes container-clock-skew as a failure mode. Items scheduled
within the next 60 s are picked when their tick arrives.

#### Inv-12: Assignee membership

`items.assignee_id`, when non-null, MUST reference a `users.id` who
is also a current `list_members.user_id` for the item's list. This
is enforced at write time by `assign_item`'s membership-scoped
resolution — there is no DB-level constraint (the schema is frozen),
so the invariant is application-enforced.

If a member is later removed from a list (Phase 3
`member_removed` flow via `DELETE
/api/lists/[id]/members/[memberId]`), the removal handler MUST also
clear any items where `assignee_id = removed_user_id` for that list,
emitting one `item_unassigned` activity row per affected item, all
inside the same transaction as the membership delete. Otherwise we
violate Inv-12.

The reminder dispatcher reads `assignee_id` defensively: if the
join finds no current `list_members` row for the assignee (a stale
assignee_id pointer that slipped past), the DM falls back to the
list owner. This is belt-and-braces; the membership-removal flow
above should already have nulled the field.

#### Inv-13: Activity log emission point for membership changes

`share_list` (invite creation) emits **no** `activity_log` row — an
invitation is not a membership change. The `member_added` row is
written at **accept time**, in the transaction that creates the
`list_members` row, with `payload_before = null` and `payload_after
= MemberSnapshot`.

Member removal (`DELETE /api/lists/[id]/members/[memberId]`) emits
one `member_removed` row (`payload_before = MemberSnapshot,
payload_after = null`) plus one `item_unassigned` row per affected
item, all in the same transaction (Inv-1).

Role changes (`PATCH /api/lists/[id]/members/[memberId]`) emit one
`member_role_changed` row with both snapshots populated; only
`role` differs between them. Role changes are owner-only.

Phase 3 only ships the `accept-invite` (member_added) and `remove
member` (member_removed) flows in the UI; `member_role_changed`
plumbing is implemented for completeness but not surfaced in the
share-sheet UI until Phase 4.

#### Inv-14: Phase-3 error codes (extend Inv-4)

| Code | When |
|---|---|
| `already_member` | `share_list` invitee is already a `list_members` row for the target list |
| `cannot_share_inbox` | `share_list` target list has `is_inbox = true` |
| `invitee_dm_failed` (warning, not error) | DM to invitee failed (403 = invitee hasn't started bot); invite row still valid |
| `not_a_member` | `assign_item` resolves a user who is not a list member |
| `assignee_ambiguous` | `assign_item` resolves multiple candidates |
| `cannot_schedule_note` | `schedule_reminder` called on `is_checkable=false` item |
| `due_at_in_past` (warning) | `schedule_reminder` or `create_item` `due_at` is in the past |
| `invite_expired` | invite-accept on a row past `expires_at` |
| `invite_already_accepted` | invite-accept on a row that already has `accepted_at` set; idempotent UI ("you're already in") |
| `invite_username_mismatch` | invite-accept where the authenticated user's `telegram_username` doesn't match `invited_username` (lowered compare) |

`invitee_dm_failed` and `due_at_in_past` are **warnings** (returned
in `output.warnings: string[]`) not errors — the executor still
returns `ok: true`. All other codes are errors.

## `list_invites` lifecycle

> Detailed contract for Backend (`src/lib/server/lists/invite.ts`,
> `src/lib/server/lists/accept-invite.ts`).

### Token format

```ts
import { randomBytes } from 'node:crypto';
const token = randomBytes(32).toString('base64url'); // 43 chars
```

256 bits of CSPRNG entropy. URL-safe (no padding `=`, no `+/`).
Stored as-is in `list_invites.token` and compared exactly (no
lower-casing). Per Inv-10.

### `expires_at` default

7 days from creation (`now() + interval '7 days'`). Set in the
INSERT statement, not the schema default — keeps schema portable.
Phase 3 does not expose a custom-expiry path; Phase 4 may.

### State machine

```
                    ┌──── accepted (accepted_at set, accepted_by_user_id set)
                    │
   pending  ────────┼──── expired   (now() > expires_at; no accepted_at)
                    │
                    └──── revoked   (Phase 4 — DELETE row; not Phase 3)
```

Phase 3 does NOT implement explicit revoke; the share-sheet UI shows
pending invites with a "remove" button that DELETEs the row, but
the spec for the backing endpoint and any audit-log emission is
deferred to Phase 4 (folded into `member_removed` semantics).

### Accept-flow transaction

`POST /api/invites/[token]/accept` runs the following in a single
Drizzle transaction (Inv-1):

1. SELECT the invite row by token; lock for update (`FOR UPDATE`
   to serialize concurrent accepts).
2. If `accepted_at IS NOT NULL` → return `invite_already_accepted`
   with `{ listId }` so the client can navigate to the list. Idempotent.
3. If `expires_at < NOW()` → return `invite_expired`.
4. Look up the calling user's `telegram_username`; lowered compare
   against `invited_username`. Mismatch → `invite_username_mismatch`.
   (Note: a user may have re-set their handle after the invite was
   created; we lock to the username on file at invite time. Phase 4
   may relax this if friction emerges.)
5. INSERT `list_members` row with `role = invite.role`, `invited_by
   = invite.invited_by`, `accepted_at = NOW()`. Unique constraint
   on `(list_id, user_id)` + the surrounding row lock from step 1
   prevent double-add.
6. UPDATE `list_invites` row: `accepted_at = NOW()`,
   `accepted_by_user_id = caller.id`.
7. INSERT `activity_log` row: `entity_type='member'`, `entity_id =
   <new list_member id>`, `action='member_added'`, `actor_id =
   caller.id`, `payload_before = null`, `payload_after =
   MemberSnapshot`.

### Idempotency

- Re-tapping the deeplink AFTER acceptance lands on
  `(app)/invites/[token]/page.tsx`; the page calls `GET
  /api/invites/[token]` which returns `InviteTokenInfo` with
  `isAccepted = true`. The client renders "You're already in {list}"
  and offers a button to navigate to the list (does NOT call accept
  again).
- Re-tapping the deeplink ON another device that's already a member
  via a separate mechanism (Phase 4 only — Phase 3 has no other
  add-member path) follows the same `invite_already_accepted` ack.

### Username case

- `users.telegram_username` is stored as Telegram delivers it
  (Telegram allows mixed case, though it's case-insensitive in
  practice).
- The `users_telegram_username_idx` index is on
  `lower(telegram_username)`.
- All comparisons in Phase 3 (share, accept, assign) use
  `lower(...)`.
- `list_invites.invited_username` is stored **already lowered**;
  no further normalization needed at compare time.

## Reminder dispatcher (cron)

> Detailed contract for Backend (`src/lib/cron/dispatch-reminders.ts`).
> Frontend has no surface here; AI has only the `schedule_reminder`
> tool path.

### Pickup query

Drives off the existing `items_due_at_idx` partial index (Phase 1):

```sql
-- Conceptually (Drizzle equivalents in dispatcher):
SELECT
  i.id           AS item_id,
  i.list_id      AS list_id,
  i.text         AS text,
  i.due_at       AS due_at,
  owner.telegram_id    AS owner_telegram_id,
  owner.locale         AS owner_locale,
  assignee.telegram_id AS assignee_telegram_id,
  assignee.locale      AS assignee_locale
FROM items i
INNER JOIN lists l       ON l.id = i.list_id
INNER JOIN users owner   ON owner.id = l.owner_id
LEFT JOIN  list_members lm
       ON lm.list_id = i.list_id
      AND lm.user_id = i.assignee_id
LEFT JOIN  users assignee
       ON assignee.id = lm.user_id   -- only joins when assignee is still a member
WHERE i.due_at <= NOW()
  AND i.reminder_sent = false
  AND i.archived_at IS NULL
LIMIT 100;
```

Result shape: `ReminderJobItem[]` (typed exactly).

The LEFT JOIN onto `list_members` then `users` enforces Inv-12
defensively: a stale `assignee_id` referencing a no-longer-member
yields `assigneeTelegramId = null`, and the dispatcher falls back
to owner.

`LIMIT 100` per tick caps a backlog burst (e.g. cron fell behind);
the next tick picks up the next 100. At Phase-3 scale (single
flagship project) we never approach this; safety belt for self-host
operators.

### Per-row dispatch

For each `ReminderJobItem`:

1. Choose target chat: `assigneeTelegramId ?? ownerTelegramId`.
2. Choose locale: `assigneeLocale ?? ownerLocale` (drives bot
   message language).
3. Compose DM via grammY: list emoji + name + item text + due-time
   (formatted in the recipient's locale + timezone). Use grammY's
   `formatter` helper for MarkdownV2 escaping (per project gotcha).
4. `bot.api.sendMessage(targetChat, body)`.
5. On success: `UPDATE items SET reminder_sent = true WHERE id = $1
   AND reminder_sent = false` (Inv-11).
6. On failure: leave `reminder_sent = false`; log warning; continue.
   No retry within the same tick (next minute will retry).

### Activity log

Reminder send is **not** a list mutation — no `activity_log` row is
written. The reminder fired is implicit in
`reminder_sent = true`. Phase 4's audit/restore F2 may add a
`reminder_sent_at` column if the operator surface needs it; out of
scope for Phase 3.

### Cron timezone

Dokploy runs the container in UTC (`TZ=UTC`). `items.due_at` is
`timestamptz`. Comparison uses Postgres' `NOW()`, which is also UTC
under the same `timestamptz` semantics — there is no timezone bug.
User timezone is **presentation-only** (renderers in bot DM body /
Mini App use `users.timezone` to format display strings).

### Heartbeat

End-of-tick optional ping to `LISTGRAM_HEARTBEAT_URL` if set:

```ts
if (env.LISTGRAM_HEARTBEAT_URL) {
  await fetch(env.LISTGRAM_HEARTBEAT_URL, { method: 'GET' }).catch(() => {});
}
```

Failure is silently swallowed — the heartbeat is monitoring, not
load-bearing for delivery. Defaults: env var unset → no-op.
Resolves the "cron heartbeat target" open question
(`handoff/specs/CLAUDE.md` § Open questions / TODOs).

### Manual testing

`npm run cron` (per `CLAUDE.md`) runs the dispatcher once locally
against `DATABASE_URL`; useful for testing without waiting for cron.

## `assign_item` mechanics — extended notes

The tool description above covers the LLM-facing contract. Backend
implementation notes:

- The username resolution query joins `list_members` to
  `users` filtered by `list_id`, then matches `lower(telegram_username) =
  $1` (with leading `@` stripped) — exact, indexed via
  `users_telegram_username_idx`.
- Fallback prefix match on `telegram_first_name` (Postgres `ILIKE
  $1 || '%'` with `escapeLike()` from `_shared.ts` for the input)
  is also list-scoped.
- The reminder dispatcher (above) DMs the assignee, not creator,
  when `assignee_id IS NOT NULL` AND the assignee is still a list
  member. If the assignee was removed from the list mid-cycle, the
  dispatcher falls back to the owner (Inv-12 defensive read).
- AI-agent's system prompt (`src/lib/ai/prompts/system.v1.ts`)
  needs a short Phase 3 addendum: when the user writes "@ali ...",
  the model should call `assign_item` rather than embedding the
  mention in the item text. Prompt update is AI-agent's
  responsibility, not Architect's; this contract just asserts the
  expectation.

## Activity feed read pattern (B1)

> Detailed contract for Backend
> (`src/app/api/lists/[id]/activity/route.ts`) and Frontend
> (`src/app/(app)/lists/[id]/activity/page.tsx`).

### Endpoint shape

`GET /api/lists/[id]/activity?limit=50&before=<cursor>`

Query params:
- `limit` — int 1..100, default 50.
- `before` — optional ISO 8601 cursor; rows with `created_at <
  cursor` returned (strict less-than for stable pagination).

### Response

```ts
{
  ok: true,
  data: {
    rows: ActivityFeedRow[];      // ordered by created_at desc
    nextCursor: string | null;    // pass back as `before=`; null when fewer than `limit` rows returned
  }
}
```

`ActivityFeedRow` shape is fixed in `src/lib/types/index.ts`:

```ts
{
  id, listId, entityType, entityId, action,
  actorId, actorFirstName, actorUsername, actorPhotoUrl,
  payloadBefore: unknown, payloadAfter: unknown,
  createdAt: string,
}
```

### Single-query join (no N+1)

```sql
SELECT
  al.id, al.list_id, al.entity_type, al.entity_id, al.action,
  al.actor_id,
  u.telegram_first_name, u.telegram_username, u.telegram_photo_url,
  al.payload_before, al.payload_after, al.created_at
FROM activity_log al
INNER JOIN users u ON u.id = al.actor_id
WHERE al.list_id = $1
  AND ($2::timestamptz IS NULL OR al.created_at < $2)
ORDER BY al.created_at DESC
LIMIT $3;
```

This rides on the existing `activity_list_recent_idx` (`(list_id,
created_at desc)`).

### Auth / membership gate

- The route requires an authenticated Better Auth session (Mini App
  middleware enforces).
- Membership check (Inv-2) extended for read: any role
  (`owner | editor | viewer`) may read the activity feed of a list
  they belong to. Non-members → `forbidden`.

### Day grouping

Day-grouping with sticky labels is **client-side**. Backend returns a
flat array; Frontend groups by `createdAt` rendered in the user's
timezone (`users.timezone`).

### Localization

Activity sentence localization (TR/EN) is **Frontend's** job. Backend
returns the raw `action` enum + actor name + payload snapshots; the
Frontend `<ActivitySentence />` component picks the localized
template based on `users.locale`.

### Real-time-ish polling

Frontend uses TanStack Query `refetchInterval: 5000` on the activity
feed view (per `agents.md` Phase 3 plan). Pause on hidden tab via
visibility API. No websockets; polling is the Phase 1+ contract.
