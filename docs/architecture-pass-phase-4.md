### P2-3: Cron heartbeat semantics — fires on partial-failure ticks

**Resolution:** Adopt the **simpler option**: keep current behavior
(heartbeat fires whenever `dispatchReminders()` returns without
throwing) AND document it as **liveness, not health**. Add a one-line
JSDoc comment above `maybePingHeartbeat` and a one-line operator note
in `.env.example` next to `LISTGRAM_HEARTBEAT_URL`.

The "alive vs healthy" split is rejected because:
- The intent of the heartbeat env var (per Phase 3 architecture pass)
  was always "is the cron container reaching Postgres". A 100%-failed
  delivery tick still proves both.
- Per-row delivery health is observable via Sentry breadcrumbs +
  the existing per-row `console.error` log. Operators with strict
  delivery SLAs add their own log-based alarms.
- A separate "healthy" ping requires a second env var and a second
  Better Stack heartbeat — doubles operator setup cost for marginal
  signal.

**Concrete change:** above `maybePingHeartbeat()`, add:

```ts
/**
 * Liveness ping — fires whenever the dispatcher loop completes without
 * throwing, regardless of per-row delivery success. The semantic is
 * "is the cron container reaching Postgres", not "is delivery healthy".
 * Per-row failures are observable via the per-item `console.error`
 * logs + Sentry breadcrumbs; configure separate log-based alarms if
 * delivery health matters operationally.
 */
```

`.env.example` line:

```
# Optional liveness heartbeat — pinged when the cron tick completes
# (NOT a delivery-health signal; see dispatch-reminders.ts).
LISTGRAM_HEARTBEAT_URL=
```

**Owner:** Backend.

### P2-5: Inv-11 "≥3 consecutive failures" warning unimplemented

**Resolution:** **Accept the gap, deprecate the in-memory streak
requirement, replace with a post-hoc detection query.** Per-tick cron
container has no in-memory state across ticks (by Dokploy design), so
the original Inv-11 contract was unimplementable. The replacement is
Inv-15 (below) — a stuck-item detection query that runs at the end of
every tick and logs `reminder_send_persistent_failure` for any item
with `due_at < NOW() - 5 minutes AND reminder_sent = false AND
archived_at IS NULL`. No DB column added (schema stays frozen). The
gap is operator-observability only; reminders still get retried
correctly per Inv-11.

**Concrete change:** rewrite the relevant Inv-11 paragraph in
`docs/architecture-pass-phase-3.md` § "Inv-11" via a footnote pointer
to Inv-15 here. (Architect-agent owns this doc; the Phase 3 doc gets a
brief amendment header pointing forward.)

**Owner:** Backend (implementation per Inv-15) + Architect (this doc).

### P2-6: N+1 in transaction

**Location flagged in review:** the `acceptInvite()` transaction in
`src/lib/db/queries/invites.ts:142-149` selects the caller for username
validation, then `:221-229` re-selects the SAME user to refine the
`MemberSnapshot.user` payload — two SELECTs on `users` for the same
`callerId` inside one transaction.

(Note: the `removeMember` per-item loop in `members.ts:225-243` was
also discussed in P2-5 of the Phase 3 review as N+1 within a
transaction. Per that review, it is **informational only** at Phase 3
scale and "Phase 3 should NOT preemptively optimize". Phase 4 also
defers the `removeMember` batch — it's a real correctness-neutral
optimization that belongs in a Phase 5+ performance pass when actual
list sizes inform the trade-off. **`removeMember` is OUT of Phase 4
scope.**)

**Resolution for `acceptInvite`:** consolidate the two SELECTs into
one, fetching `id`, `telegramUsername`, `telegramFirstName`,
`telegramPhotoUrl` in a single round-trip. Use the result for both the
username check and the snapshot's `user` field. Remove the second
SELECT entirely.

```ts
// src/lib/db/queries/invites.ts — replace lines 142-163 + 221-237
const [caller] = await tx
  .select({
    id: users.id,
    telegramUsername: users.telegramUsername,
    telegramFirstName: users.telegramFirstName,
    telegramPhotoUrl: users.telegramPhotoUrl,
  })
  .from(users)
  .where(eq(users.id, callerId))
  .limit(1);
if (!caller) {
  return { ok: false, code: "not_found", message: "Caller not found." };
}
const callerLowered = (caller.telegramUsername ?? "").toLowerCase();
// ...validation...
// Then later, build snapshot.user directly from `caller`:
snapshot.user = {
  telegramFirstName: caller.telegramFirstName,
  telegramUsername: caller.telegramUsername,
  telegramPhotoUrl: caller.telegramPhotoUrl,
};
```

**Owner:** Backend.

### A3 — Forwarded message → action items

**Spec:** Forwarding a Telegram message to the bot triggers item
extraction. The webhook handler detects `update.message.forward_origin`
(grammY) and routes the text through a dedicated forwarded-message
prompt template (`src/lib/ai/prompts/forwarded.ts`) instead of the
general-purpose conversation prompt. The LLM extracts an array of
discrete action items, then calls `create_item` per item using the
existing tool path. The bot's reply summarizes what was created
("3 öğe Inbox'a eklendi: …"). Forwarded messages from channels and
users are both supported; sender attribution is included as part of
the prompt context but NOT stored in `items.text`.

**Owner:** Backend (forward detection in webhook router) + AI-agent
(forwarded prompt template).

**Files:** `src/lib/server/bot/handle-message.ts` (extended;
`forward_origin` detection branches to the forwarded path),
`src/lib/ai/prompts/forwarded.ts` (NEW).

**Acceptance:**
- Webhook detects `forward_origin` field
- LLM extracts items from forwarded text
- Items added to user's chosen/inferred list (Inbox by default; LLM
  may resolve a list via the existing `create_item` rules if the
  forwarded text mentions one)

**New invariants:** Inv-16 (forwarded extraction caps; see § 5).

### D3 — Schedule-a-message pickup

**Spec:** Telegram natively supports scheduled messages — a user can
schedule a DM to the bot at a future time, and Telegram delivers it
when the time arrives. The webhook handler is already time-agnostic

### Inv-15: Reminder persistent-failure detection (replaces Inv-11 streak)

After the per-row dispatch loop in `dispatchReminders()` completes,
the dispatcher runs a single SELECT:

```sql
SELECT id, list_id, due_at
FROM items
WHERE due_at < (NOW() - interval '5 minutes')
  AND reminder_sent = false
  AND archived_at IS NULL
LIMIT 50;
```

For each row returned, the dispatcher logs
`console.warn('reminder_send_persistent_failure', { itemId, listId, dueAt })`.
No `reminder_sent` flip; operator must intervene (most likely the bot
token is revoked or the user blocked the bot).

Cap of 50 in the post-hoc query is a logging-noise cap; if the
operator's bot is fully broken there could be thousands of stuck
items, and we don't want to flood logs.

The SELECT runs OUTSIDE the per-row try/catch. A failure to run the
post-hoc query swallows (caught + logged) — it must never crash the
dispatcher.

### Inv-16: Forwarded-message extraction caps

The forwarded-message LLM path extracts at most **20 items per
forward**. Cap enforced in two places:

1. AI-agent's `forwarded.ts` prompt instructs the model: "extract at
   most 20 distinct action items".
2. Backend's `handle-message.ts` forwarded branch limits the tool-call
   loop to ONE round-trip in the forwarded path (the model sees the
   forwarded text + must emit all `create_item` calls in a single
   turn). Subsequent turns revert to the standard 5-round-trip cap if
   the user follows up conversationally.

Forwarded message text > 6000 chars is truncated server-side before
being passed to the LLM (head-truncated to preserve the start), with
a note in the prompt: "(message truncated)".

### Inv-17: Config-completeness check

Mini App `middleware.ts` (extended) reads `env.BOT_TOKEN`,
`env.WEBHOOK_SECRET`, `env.DATABASE_URL` at request boundary. If any
is unset OR `DATABASE_URL` connection check fails, redirect to
`/app/setup` (C2 wizard). Once all three are healthy AND the wizard
has been completed (track via a one-row `app_state` mechanism — OR
just a probe query — Backend's call), the redirect drops out.

Production listbull.org has these set at deploy time, so the redirect
never fires there. Self-host operators see the wizard on first load.

### Inv-18: Snapshot URL HMAC signing

Snapshot URLs sign `${listId}:${exp}` (where `exp` is unix-ms) using
HMAC-SHA256 with `ENV_KEY` (or a dedicated `SNAPSHOT_SIGNING_KEY` —
Backend chooses; if separate, document in `.env.example`). The HMAC
output is base64url-encoded.

The `(marketing)/snapshot/[id]` page:
1. Reads `?exp=` and `?token=` query params.
2. Verifies `exp > now()` (rejects expired with 410 Gone).
3. Recomputes the HMAC and constant-time compares with `?token=`
   (rejects mismatch with 404).
4. Loads the list by `id` and renders.

URLs are NOT re-issued on access — operators wanting a "live" snapshot
generate a new URL via the share sheet. This keeps the URL stateless
(no DB column tracking re-issuance).

### Inv-19: Locale catalog completeness

Every key in `messages/tr.json` MUST exist in `messages/en.json` and
vice-versa. Reviewer adds a CI script (`scripts/check-locale-keys.ts`
OR a Vitest test under `tests/unit/`) that diffs the two catalogs and
fails CI on missing keys. The script is owned by Reviewer; Frontend
keeps catalogs in sync as features ship.

### Inv-20: Export caller-only filter

The `ExportBundle` returned by `GET /api/settings/export` includes:

- `lists`: every list where `caller_id` is in `list_members` for that
  list (any role) — owners, editors, and viewers see lists they
  belong to.
- `items`: every item in the included lists where `caller_id ===
  items.created_by` OR `caller_id === assignee_id` — the caller's
  own contributions only. Other members' items are NOT exported.
- `activity`: rows where `caller_id === activity_log.actor_id` —
  the caller's own actions only.
- `messages`: rows where `caller_id === messages.user_id` — the
  caller's own conversation history.

NEVER included: encrypted API key (`openrouter_api_key_encrypted`),
session cookies, other users' messages, other users' activity, other
users' items even in shared lists.

The route handler is the SOLE enforcement point; no other endpoint
returns these JOINs. Audit Reviewer-agent runs grep against
`src/lib/server/export.ts` to confirm the filter clauses.

### Inv-21: Restore window enforcement

`POST /api/lists/[id]/restore` body schema:
`{ activityLogId: string }`. The handler:

1. Looks up the activity_log row by id.
2. Verifies `action === 'item_deleted'`.
3. Verifies `(now() - created_at) <= 30 days` — rejects with
   `restore_window_expired` otherwise (Phase 4 error code, extends
   Inv-4).
4. Verifies the caller is the list's owner (Inv-2 owner-only).
5. Inside a single transaction (Inv-1):
   - Reads `payload_before` as `ItemSnapshot` (zod-validate at the
     boundary; reject `restore_payload_invalid` if malformed).
   - INSERTs new `items` row from the snapshot. Drops `id` (new uuid),
     keeps everything else; sets `archived_at = null`,
     `reminder_sent = false` if `due_at` is in the future, else true.
   - INSERTs new `activity_log` row with `action='item_created'`,
     `payload_before = null`, `payload_after = ItemSnapshot` of new
     row. (Phase 5+ may distinguish `item_restored` as a separate
     action — out of scope here; the `payload_after` shape is
     identical.)

The 30-day window is enforced server-side regardless of UI state — UI
shows the Restore button only on rows ≤30 days old (via
`AuditEntryWithRestore.canRestore`), but the server re-checks for
defense-in-depth.
